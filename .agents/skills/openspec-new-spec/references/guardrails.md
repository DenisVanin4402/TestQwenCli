# Guardrails

## КРИТИЧНО — роль документа

- Спецификация — документ **системного аналитика**, а не разработчика.
- Описывай ЧТО сервис делает, КАК себя ведёт, КАКИЕ данные принимает/отдаёт — но **НИКОГДА** не пиши КАК реализовать в коде.
- **Критерий самодостаточности**: по одной спеке разработчик должен иметь возможность **с нуля восстановить сервис** без доступа к текущему коду. Если для понимания раздела нужно лезть в исходники — раздел неполный.

### ЗАПРЕЩЕНО включать в спецификацию

- Код на любом языке (Java, Python, Go и т.д.).
- Реализацию методов, классов, функций.
- Unit-тесты, integration-тесты, тестовый код.
- Конкретные алгоритмические решения (это задача разработчика).
- **Ссылки на объекты исходного кода**: имена классов (`OrderService`, `OrderController`), методов (`createOrder()`), пакетов (`com.shop.orders.service`), бинов, модулей, переменных.
- **Ссылки на файлы исходного кода**: `src/main/java/...`, `pkg/handlers/...`, `app/services/...` и подобные пути к реализации.

### ПРАВИЛЬНО

- Бизнес-правила (2.3) — каждое правило ОБЯЗАНО содержать ДВЕ части: (1) ФОРМУЛИРОВКУ в формате ЕСЛИ / ТОГДА / ИНАЧЕ и/или математики; (2) ПОШАГОВЫЙ БИЗНЕС-ПСЕВДОКОД — нумерованные шаги с явной обработкой краёв (null, пустой ввод, округления, гонки). Псевдокод — на языке, независимом от реализации (без Java/Python/Go-синтаксиса, без аннотаций фреймворков, без обращений к коллекциям, потокам, библиотекам, кешам, индексам БД). Технический псевдокод (про блокировки, потоки, кеши, индексы БД) — запрещён, это территория архитектора.
- **Бизнес-формулы** (расчёт сумм, скидок, тарифов, рейтингов, штрафов и т.п.) — частный случай правила 2.3: математическое выражение + пошаговый псевдокод с округлением и обработкой краевых значений.
- **Обработчики по триггерам (раздел 2.1.2.N)** — ФИКСИРОВАННЫЙ скелет для КАЖДОГО триггера сервиса: Триггер, Инициатор, Вход, Предусловия (guard), Шаги обработки (нумерованные, со ссылками на 2.3 и 2.4), Side-effects, Выход/ответ, Ошибочные ветки, Идемпотентность (ключ + поведение при повторе). Допустимо указывать транзакционную границу на бизнес-уровне («атомарно в одной транзакции», «outbox», «saga») — без упоминания конкретных фреймворков/библиотек.
- **Типовой флоу клиента (раздел 2.1.1)** — таблица последовательности API-вызовов клиента. Отвечает на вопрос «что клиент дёргает подряд». Для чисто событийных сервисов — явная пометка «Не применимо».
- **FSM (раздел 2.4)** — таблица переходов (from, to, trigger, guard, side-effect), список состояний с семантикой, диаграмма (PlantUML/Mermaid). Guard-условие должно ссылаться на правило из 2.3, а не содержать решающую логику внутри FSM.
- **Глоссарий (1.2)** — таблица термин/синонимы/определение. Каждый доменный термин, появляющийся в спеке, обязан быть здесь.
- **Бизнес-инварианты (1.3)** — нерушимые правила в формулировке ВСЕГДА / НИКОГДА. Каждый инвариант наблюдаем (можно проверить тестом или запросом).
- Модели данных как таблицы полей (имя, тип, обязательность, пример, описание).
- API как описание эндпоинтов (метод, путь, request/response JSON-схема).
- Интеграции как описание взаимодействий (протокол, формат, таймаут, retry).
- Рекомендации к критериям приёмки в формате КОГДА / ТОГДА (без тестового кода).
- Конфигурация как таблица параметров (имя, тип, default, описание).
- **Перекрёстные ссылки**: каждый шаг сценария 2.1/2.2 ссылается на правило 2.3.X и/или переход 2.4.X. Каждый критерий приёмки 3 ссылается на переход FSM или инвариант, который проверяет.

### Допустимые ссылки на артефакты-контракты

Спека **может и должна** ссылаться на контрактные артефакты — они часть описания поведения:

- **OpenAPI/Swagger**: `openapi/order-api.yaml`, `swagger/payment.json`.
- **Protobuf**: `proto/payment_service.proto`.
- **Avro**: `schemas/order-created-event.avsc`.
- **JSON Schema**: `schemas/CreateOrderRequest.json`.
- **GraphQL SDL**: `schema.graphql`.
- **SQL DDL/migrations**: `migration/V1__create_orders.sql`, `db/changelog.yaml`.
- **XSD / WSDL**: `contracts/order.xsd`.
- **Диаграммы**: PlantUML (`diagrams/order-flow.puml`), Mermaid, C4, sequence/activity.

**Ключевое отличие**: контракт описывает **поведение интерфейса** и самодостаточен. Класс/метод/файл исходного кода описывает **реализацию** и без кода бесполезен.

### Допустимые форматы схем

- **SQL** — ТОЛЬКО как рекомендации по DDL/DML (CREATE TABLE, ALTER TABLE) для описания структуры данных.
- **JSON-schema**, **Protobuf**, **Avro** — допустимы для описания моделей данных.
- **Бизнес-псевдокод и математические формулы** — допустимы в 1.3 и 2.3 для описания вычислительной части бизнес-логики (без привязки к языку реализации, без обращений к коллекциям, потокам, библиотекам, кешам).
- **Код приложения** (Java, Python, Go и т.д.) — ЗАПРЕЩЁН полностью.

## Процесс

- **НИКОГДА** не создавай спецификацию, пока остаются неясности — проведи полноценное интервью до полной ясности.
- Перед созданием файла подведи итог ВСЕХ данных о сервисе и получи явное подтверждение пользователя.
- **ВСЕГДА** читай шаблон `../templates/service-spec.md` из состава скила.
- Удали ВСЕ HTML-комментарии из результата.
- Каждый раздел должен содержать реальный контент — никаких остатков `<!-- placeholder -->`.
- Проверь, что файл существует после записи.
