## 1. Требования к логированию событий

### 1.1 Уровни логирования

Уровень события (Level) – один из заранее заданных уровней
важности события, выделяются следующие уровни:

- **OFF** - Логирование отключено. Никакие логи не записываются, все
  будут проигнорированы.
- **ERROR** - ошибка нормального функционирования, означающая отказ в
  обработке запроса и требующая реакции со стороны сопровождения. Может
  включать в себя: Непредусмотренные неверные сочетания параметров и
  условий, которые привели к невозможности обработать запрос и выдать
  логически корректный ответ Отсутствие требуемых ресурсов
  (недоступность системы хранения данных или внешних систем).
- **WARN** - возможная проблема в работе системы, которая может означать
  наличие других неявных проблем. Обычно не требует немедленной реакции
  со стороны поддержки. Может включать в себя:
    - Отказ в обработке всего или части запроса из-за неверных параметров
      (предусмотренные проверки на неверные сочетания параметров и условий),
      если в обычном случае параметры вызова должны генерироваться
      автоматически и такие неверные сочетания указывают на проблемы в
      сопряжённой системе;
    - Отказ в обработке из-за отсутствия у пользователя необходимых разрешений (не
      заменяет собой запись в журнале аудита и безопасности).
- **INFO** - «обычная» запись логирования событий, ожидаемые действия
  системы.
- **DEBUG** – запись в журнале событий, предназначенная для разбора
  проблемных ситуаций. Обычно уровень DEBUG отключён на
  production-системах, но включён на стендах тестирования для
  упрощения поиска ошибок. Может включать в себя самую разную информацию
  на усмотрение разработчика.
- **TRACE** - ещё более детальный по сравнению с DEBUG уровень. Обычно
  никогда не включается для всей системы, так как может привести к
  попытке регистрации каждого действия системы (в т.ч. одна строка кода
  = одно событие). Используется разработчиками на локальных стендах для
  локализации уже найденных проблем, один из инструментов отладки
  программы. Включается только на стендах тестирования.
- **ALL** - уровень, при котором будут записаны все логи из системы.

### 1.2 Настройка логирования

Процесс логирования настраивается через конфигурационный файл
и имеет несколько иерархичных уровней (Level):

- **DEBUG** - фиксируются события с уровнями: DEBUG, INFO, WARNING, ERROR;
- **INFO** - фиксируются события с уровнями: INFO, WARNING, ERROR;
- **WARN** - фиксируются события с уровнями: WARNING, ERROR;
- **ERROR** - фиксируются события с уровнями: ERROR;
- **TRACE** или **ALL** - все уровни.

Для формирования логов журнала используется библиотека [logback](https://logback.qos.ch/).

При формировании прикладного лога необходимо исключить возможность
попадания в него конфиденциальной информации K1/K2 (номера карт, номера
ДУЛ, фамилии, адреса и контактные данные клиентов и т.п.). В случае
наличия таких данных они должны "маскироваться" символами "*" перед
сохранением в лог.

## 2. Требования к кодам ошибок

### 2.1 Система кодирования

Коды должны следовать единой системе нумерации:

- 1000-1999: Коды ошибок (ERROR)
- 2000-2999: Предупреждения (WARN)
- 3000-3999: Информационные сообщения (INFO)
- 4000-4999: Отладочная информация (DEBUG)

### 2.2 Структура кода

X Y Z
│ │ └── Уникальный идентификатор события (00-99)
│ └────── Функциональный модуль (0-9)
└──────── Тип события (1-ERROR, 2-WARN, 3-INFO, 4-DEBUG)

### 2.3 Зарезервированные диапазоны по модулям

| Модуль                        | Диапазон ERROR | Диапазон INFO | Описание модуля            |
|-------------------------------|----------------|---------------|----------------------------|
| 0 - Общие                     | 1000-1099      | 3000-3099     | Общесистемные ошибки       |
| 1 - Аутентификация            | 1100-1199      | 3100-3199     | Модуль auth-service        |
| 2 - Авторизация               | 1200-1299      | 3200-3299     | Проверка прав доступа      |
| 3 - Взаимодействие с агентами | 1300-1399      | 3300-3399     | Проброс сообщений агентам  |
| 4 - Внешние интеграции        | 1400-1499      | 3400-3499     | Email, SMS, внешние API    |
| 5 - 9 - Резерв                | 1500-1999      | 3500-3999     | Для будущего использования |

### 2.4 Обязательные требования

- **Уникальность**: Каждый код должен быть уникальным в пределах системы;
- **Консистентность**: Коды одного типа и модуля должны быть сгруппированы;
- **Расширяемость**: При добавлении новых модулей использовать зарезервированные диапазоны;
- **Документирование**: Все коды должны быть зарегистрированы в этом документе;

## 3. Требования к кодам ошибок HTTP

Данный раздел определяет единый формат для возврата ошибок в HTTP API на основе
стандарта [RFC 9457 (ранее RFC 7807)](https://datatracker.ietf.org/doc/html/rfc9457) c целью обеспечения консистентности
и упрощения
обработки ошибок на стороне клиента.

* *Основной код состояния HTTP должен быть установлен в заголовке ответа.*
* *В теле ответа в поле **status** (если оно присутствует) должен быть указан код состояния HTTP `аналогичный` коду
  состояния HTTP в заголовке ответа.*
* *Для ответа используется application/problem+json*
* *В теле ответа используется JSON-объект в формате "Problem Detail*

### 3.3. HTTP-статусы ошибок AI Risk Hub

| Код   | Событие                           | Формат сообщения      |
|-------|-----------------------------------|-----------------------|
| `400` | Неправильный, некорректный запрос | Bad Request           |
| `403` | Недостаточно прав                 | Forbidden             |
| `404` | Ресурс не найден                  | Not Found             |
| `500` | Внутренняя ошибка сервера         | Internal Server Error |

### 3.2 Базовые атрибуты ответа кода ошибки

| Атрибут     | Тип данных | Обязательность | Описание                                                                                                                                                                          |
|-------------|------------|----------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `type`      | string     | [1]            | Ссылка URI, идентифицирующая тип проблемы.`Рекомендуется использовать относительные URI, начинающиеся с /problems/`                                                               |
| `title`     | string     | [1]            | Краткое, читаемое человеком резюме типа проблемы. <br/>Оно не должно меняться между несколькими случаями одного и того же типа проблемы, <br/>за исключением случаев локализации. |
| `status`    | number     | [1]            | Код статуса HTTP, сгенерированный исходным сервером. `Должен быть аналогичным коду статуса HTTP в заголовке ответа.`                                                              |
| `errorCode` | string     | [1]            | `Расширение` Человекочитаемый код ошибки                                                                                                                                          |
| `detail`    | string     | [0-1]          | Человекочитаемое описание возникновения этой конкретной проблемы.                                                                                                                 |
| `instance`  | string     | [0-1]          | URI, идентифицирующий ресурс, связанный с этим конкретным возникновением проблемы.                                                                                                |

**Примечание:** В соответствии с **RFC 9457**, если поле `type` в ответе об ошибке отсутствует или пустое, его значение
автоматически считается равным `"about:blank"`.
Это означает, что ошибка не имеет специальной семантики помимо стандартного HTTP-статуса.
Рекомендуется всегда явно указывать `type` — либо URI, либо `"about:blank"`.

## 4. Требования к наименованию сервисов

1. Правила наименования сервисов: `{*краткое уникальное наименование
   сервера латинскими буквами*}-service в kebab-case`, например
   `email-mcp-service`.
   Уникальное наименование сервера должно отражать его логическую функцию.
