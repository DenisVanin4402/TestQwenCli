Прочитай этот документ целиком — он описывает OpenSpec workflow: какие скиллы, когда их запускать, какие статусы у change-запросов, каких правил держаться и как маппить естественно-языковые запросы пользователя на конкретный скилл.

После прочтения ничего создавать и записывать не нужно. Задача скилла — насытить твой контекст знанием о workflow. Применяй его при дальнейшей работе и сразу подскажи пользователю следующий логичный шаг (какой скилл запустить).

## Структура

- `openspec/specs/<service-name>/<service-name>.md` — актуальная спецификация сервиса (source of truth).
- `openspec/changes/<name>/` — активные change-запросы:
  - `change.md` — предложение системного аналитика;
  - `design.md` — технический проект (опционально для простых CR);
  - `tasks.md` — план задач на реализацию.
- `openspec/changes/archive/YYYY-MM-DD-<name>/` — завершённые change-запросы.

Директории создаются самими скиллами через `mkdir -p` — отдельный bootstrap не требуется.

## Скиллы

| Скилл                     | Назначение                                                                                     |
|---------------------------|------------------------------------------------------------------------------------------------|
| `openspec-teach`          | Этот учебник                                                                                   |
| `openspec-explore`        | Structured research кодовой базы: параллельные read-only субагенты по ролям (api/data/integrations/…), агрегация в `openspec/**/.research/`. Standalone — прямой вызов для разового картирования. Также работает prompt-library для `new-spec` / `propose` (они переиспользуют `openspec-explore/references/research-roles.md` и `openspec-explore/references/invocation-contract.md`, оркестрируя субагентов сами) |
| `openspec-new-spec`       | Создать спецификацию сервиса с нуля                                                            |
| `openspec-propose`        | Создать `change.md` (статус `На согласовании`)                                                 |
| `openspec-design`         | Создать `design.md` + `tasks.md` из согласованного change                                      |
| `openspec-implement`      | Выполнить `tasks.md`; обязательная верификация (сборка/тесты/линтер) перед объявлением готовности |
| `openspec-apply-change`   | Влить `change.md` в `spec.md` (откат — через git)                                              |
| `openspec-archive-change` | Переместить change в `archive/YYYY-MM-DD-<name>/`                                              |

## Жизненный цикл изменения

1. **Первая спецификация сервиса** — скилл `openspec-new-spec`.
2. **Предложение на изменение** — `openspec-propose` создаёт `change.md` (статус `На согласовании`).
3. **Ревью и согласование** — merge PR, статус вручную меняется на `Согласовано`.
4. **Технический проект** — `openspec-design` создаёт `design.md` + `tasks.md` (для сложных CR).
5. **Реализация** — `openspec-implement` выполняет задачи из `tasks.md`, ставит статус `В реализации`, по завершении запускает верификацию (сборка/тесты/линтер).
6. **Влитие в спецификацию** — `openspec-apply-change` мержит `change.md` в целевой `spec.md`, ставит статус `Реализовано`.
7. **Архивация** — `openspec-archive-change` перемещает change в `archive/YYYY-MM-DD-<name>/`. Если статус был `Реализовано` — меняется на `Архивировано`. Если другой (архивация «с warning») — физически перемещается, но статус в шапке сохраняется как был, для аудита незавершённой работы.

## Статусы change.md

| Статус          | Кто ставит                  | Когда                                                                                                                                                     |
|-----------------|-----------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------|
| На согласовании | `openspec-propose`          | Change создан, идёт ревью в PR                                                                                                                            |
| Согласовано     | Аналитик вручную            | PR с change.md вмержен, change одобрен                                                                                                                    |
| В реализации    | `openspec-implement`        | Первый запуск implement, идёт кодинг по tasks.md                                                                                                          |
| Реализовано     | `openspec-apply-change`     | Изменения вмержены в спецификацию                                                                                                                         |
| Архивировано    | `openspec-archive-change`   | Change перемещён в архив из статуса `Реализовано`. При архивации с warning (другой статус) — физически перемещено, но статус в шапке сохранён для аудита |

Обратные переходы (например, возврат из `В реализации` в `Согласовано` для правок) — только вручную.

## Резолв путей bundle

Скилы OpenSpec ссылаются на собственные `templates/*` и `references/*` — файлы рядом с их SKILL.md. Пути в SKILL.md всегда относительные к директории своего скила.

По Agent Skills Spec runtime должен передавать абсолютный путь skill-директории в обёртке активации (строка `Skill directory: <abs-path>`). Если эта строка не передана — один раз `Glob("**/openspec-skills/skills/<skill-name>/<file>")` и бери первый результат. Пусто → спроси пользователя путь установки, не ищи вручную по `~`/cwd.

## Правила для агента

- **Не редактируй спецификации сервисов напрямую** — любое изменение оформляется через `change.md` + ревью.
- **Откат в apply-change — через git** (`git checkout HEAD -- <spec>`), не через `.bak`-файлы.
- **Верификация в implement обязательна** — запуск сборки/тестов/линтера перед объявлением реализации завершённой.
- **Проси подтверждение** через AskUserQuestion tool при неоднозначных ситуациях (нечёткий статус, несколько активных changes, несколько git-репозиториев рядом). Если AskUserQuestion недоступен в клиенте — задай тот же вопрос обычным текстом в ответе и дождись ответа пользователя перед продолжением. Не угадывай.
- **Не копируй примеры из шаблонов** в итоговые файлы — используй как ориентир формата.
- **Для размытого обсуждения/обдумывания** — используй обычный чат **без скила**. Conversational-режим в `explore` отключён.
- **`openspec-explore` standalone — с чёткими параметрами** (`target=spec|change`, `service`, `anchor` для change). Напрямую — только для разового картирования сервиса или зоны изменения. Внутри `new-spec`/`propose` его код не запускается — эти скилы переиспользуют его references-файлы и оркестрируют субагентов сами.
- **Tool запуска субагента** — имя и `subagent_type` зависят от harness'а и регистрочувствительны. Таблица известных имён + алгоритм обнаружения (регекс по списку доступных инструментов) — в `openspec-explore/references/invocation-contract.md § 3.1`. `task_id` не используется — запуск one-shot.

## Приоритет инструментов анализа кода

От наиболее семантического к наименее:

1. **Serena (MCP)** — `find_symbol`, `get_symbols_overview`, `find_referencing_symbols`. Семантический анализ, точная навигация по символам.
2. **LSP-инструменты** — go-to-definition, find-references, workspace symbols (если поднят языковой сервер).
3. **Code index / embeddings** — семантический поиск по кодовой базе.
4. **Другие MCP-серверы** — любые специализированные инструменты анализа.
5. **Grep / Glob** — текстовый поиск. Последний вариант, когда семантика недоступна.

Не ограничивайся чтением отдельных файлов — строй полную картину: зависимости, вызывающий код, связанные сущности.

## Маппинг запросов пользователя на скилл

Слушай формулировки пользователя и **проактивно предлагай соответствующий скилл** (или сразу запускай, если однозначно). Примеры — не исчерпывающий список, ориентируйся на смысл.

| Скилл                     | Триггер-формулировки пользователя                                                                                                                                                          |
|---------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `openspec-new-spec`       | «напиши/опиши спецификацию сервиса», «задокументируй сервис X», «нужна спека на Y», «давай опишем новый сервис», «составь документ аналитика для Z»                                         |
| `openspec-propose`        | «давай добавим/поменяем/изменим/уберём X в сервисе Y», «нужно обновить спеку Y», «внеси изменение Z», «предлагаю переделать W», «оформим CR», «сделаем change-запрос»                        |
| `openspec-design`         | «давай планировать реализацию», «нужен техпроект/дизайн», «распиши задачи», «сделай план разработки», «подготовь tasks.md», «спроектируй как это делать»                                     |
| `openspec-implement`      | «давай кодить/реализовывать», «выполни задачи из tasks», «пиши код по плану», «реализуй change», «делай то что в tasks.md»                                                                   |
| `openspec-apply-change`   | «влей change в спеку», «обнови спецификацию по изменению», «смержи change в spec», «применить change к спеке», «зафиксируй изменение в основной спецификации»                                |
| `openspec-archive-change` | «заархивируй change», «перенеси в архив», «убери change в archive/»                                                                                                                        |
| `openspec-explore`        | «картируй сервис X», «structured research по сервису», «research codebase», «собери карту зоны изменения» — всегда с явными параметрами (target + service [+ anchor])                       |
| `openspec-teach`          | «как работает openspec?», «научи пользоваться», «что за процесс?», «объясни workflow», «какой скилл использовать?»                                                                         |

**Правила маппинга:**

- «Новый сервис/документ с нуля» → `new-spec`. «Правка существующего сервиса» → `propose`. Не путай.
- Если есть согласованный `change.md` и пользователь говорит «пора кодить» — проверь наличие `design.md`/`tasks.md`. Нет — `design`. Есть — `implement`.
- Если пользователь сомневается / формулирует вопрос неопределённо — уточни через AskUserQuestion, а не угадывай.
- Запросы вида «посмотри», «изучи», «обдумай», «обсудим варианты» — это **обычный чат без скила**, не `explore`. `explore` запускается standalone только для structured research с явными параметрами. Внутри `new-spec`/`propose` — не запускается как отдельный скил, они работают с его references-файлами напрямую.
- Если пользователь говорит «давай делать X», но в проекте нет ни одной спецификации в `openspec/specs/` — скорее всего нужен сначала `new-spec`.

## Следующий шаг

Подскажи пользователю подходящий скилл исходя из текущего состояния проекта:

- нет ни одной спецификации сервиса в /openspec → `openspec-new-spec`;
- нужно оформить изменение существующей спецификации → `openspec-propose`;
- нужно получить разовую карту сервиса или зоны изменения (с параметрами) → standalone `openspec-explore` (NB: внутри `new-spec`/`propose` ресеч идёт через переиспользование его references, не через повторный запуск скила);
- есть согласованный change, пора проектировать → `openspec-design`;
- готов `tasks.md`, пора кодить → `openspec-implement`;
- реализация вмержена, пора обновить спецификацию → `openspec-apply-change`;
- change реализован и влит, пора архивировать → `openspec-archive-change`.
