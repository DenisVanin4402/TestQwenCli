---

Влить изменения из `change.md` в основную спецификацию (Analyst Merge).

**Input**: Опционально — имя change. Если не указано, определить автоматически или предложить выбор.

**Bundle-пути.** `references/X` — относительно директории этого скила. Если harness даёт строку `Skill directory: <abs>` в обёртке активации — используй её. Иначе один раз `Glob("**/openspec-skills/skills/openspec-apply-change/references/<file>")` и возьми первый результат. Пусто → спроси пользователя путь установки, не ищи вручную.

**Bundled materials** (относительно этого SKILL.md):

- `references/merge-workflow.md` — детальная механика мержа: таблица соответствия разделов, перенумерация, пересборка TOC, семантический diff заголовков, правила отката через git, форматирование HTML-разметки.

**AskUserQuestion fallback**: если инструмент недоступен в клиенте — задай тот же вопрос обычным текстом и дождись ответа пользователя. Не угадывай.

---

## Steps

### 1. Выбрать change

Если имя указано — использовать. Иначе:

```bash
ls openspec/changes/ 2>/dev/null
```

- Один активный change (исключая `archive/`) → автовыбор.
- Несколько → выбор через **AskUserQuestion tool**.

Объяви: `Применяю change: <name>`.

### 2. Проверь наличие change.md и его статус

Прочитай `openspec/changes/<name>/change.md`.

**Если файл не существует** → предложи сначала запустить `openspec-propose`.

Проверь поле `> **Статус**:` в шапке:

| Статус | Действие |
|--------|----------|
| `Согласовано` / `В реализации` | OK, можно мержить |
| `На согласовании` | **БЛОК**: change не согласован. Попроси дождаться merge PR и ручной установки статуса `Согласовано`. Выйди. |
| `Реализовано` | Поздравь, предложи запустить `openspec-archive-change`. Выйди. |
| `Архивировано` | Сообщи, что change уже заархивирован. Выйди. |
| Другой / отсутствует | Предупреди и спроси подтверждение через **AskUserQuestion tool**. |

### 3. Прочитай контекст

Прочитай:
- `openspec/changes/<name>/change.md` (полностью).
- Целевую спецификацию — путь из шапки change.md, поле «Целевая спецификация».

Если поле пустое или файл спеки не существует:

- Пустое / `отсутствует` → предложи через AskUserQuestion: (a) запустить `openspec-new-spec` для создания спеки сначала и потом вернуться сюда, (b) указать путь вручную. По умолчанию — (a).
- Указан, но файл не существует → сообщи пользователю о рассинхроне: в change.md указан путь, но файла нет. Предложи исправить путь или создать спеку.

### 4. Проверь git-состояние целевой спеки

Точка отката — через git, не через `.bak`. Запусти:

```bash
git status --porcelain <spec-path>
```

- Файл имеет незакоммиченные изменения → попроси пользователя сначала закоммитить или откатить. Выйди.
- Файл чист → продолжай. Точка отката лежит в `HEAD`.

Детали — `references/merge-workflow.md § 1`. `.bak`-файлы создавать запрещено.

### 5. Мерж разделов change.md → спецификация

Для каждого раздела change.md, значение которого **не** `Нет изменений.`, найди соответствующий раздел в спеке **по названию, не по номеру**.

Полная таблица соответствия разделов, правила перенумерации подразделов, пересборка TOC, сохранение HTML-разметки — `references/merge-workflow.md § 2–6`.

Порядок операций:

1. Найди целевой раздел по названию. Если нет — создай в логически подходящем месте (порядок секций в шаблоне спеки — ориентир; см. `merge-workflow.md § 3` с примерами иерархии разделов).
2. Вставь контент change.md в раздел.
3. Перенумеруй все подразделы в затронутой секции.
4. Обнови внутренние ссылки (`см. п. X.Y`).
5. Пересобери TOC по всем заголовкам.
6. Инкрементируй версию спеки в шапке (напр. `1.0` → `1.1`).
7. **Сохраняй** существующий контент, не затронутый изменением.
8. **Сохраняй** HTML-разметку (цветовые маркеры `<font>...`) в тексте спеки.

### 6. Обнови статус change.md

Замени строку `> **Статус**:` в шапке на:

```
> **Статус**: Реализовано
```

Меняй ТОЛЬКО эту строку.

### 7. Запиши обновлённую спецификацию

Сохрани весь результат шага 5 (мерж разделов + обновлённый TOC + инкремент версии) **одной записью** по пути `<spec-path>` (поле «Целевая спецификация» из шапки change.md). Шаг 6 — отдельное редактирование `change.md`, не смешивай с записью спеки.

### 8. Проверь мерж — семантический diff заголовков

Используй временные файлы **внутри репозитория** (не `/tmp`) — так портируется на любой POSIX shell, включая окружения без доступа к `/tmp`:

```bash
git show HEAD:<spec-path> | grep -E '^(##|###|####) ' > .headings-before.tmp
grep -E '^(##|###|####) ' <spec-path> > .headings-after.tmp
diff .headings-before.tmp .headings-after.tmp
rm -f .headings-before.tmp .headings-after.tmp
```

Детали альтернатив и process substitution — `references/merge-workflow.md § 7`.

Убедись:
- Ни один заголовок `##/###/####` из `HEAD` не исчез (кроме REMOVED-секций).
- Нумерация подразделов последовательная.
- TOC соответствует фактическим заголовкам.
- Внутренние ссылки (`см. п. X.Y`) валидны.

Если проверка провалилась — откати через `git checkout HEAD -- <spec-path>`, исправь, повтори.

## Output

```
## Apply Complete (Analyst Merge)

**Change:** <change-name>
**Target:** <spec-path>
**Version:** X.Y → X.Y+1

### Merged Sections
- ✓ Бизнес-логика → 2.3
- ✓ Модели данных (3 ADDED) → 2.6
- ✓ Интеграции (2 ADDED) → 2.5
- ○ Безопасность — Нет изменений
...

Откат: `git checkout HEAD -- <spec-path>` (или `git diff <spec-path>` для просмотра изменений).
Готово к архивации: запусти скил `openspec-archive-change`.
```

## Guardrails

- ВСЕГДА читай change.md и целевую спецификацию перед началом.
- ВСЕГДА проверяй чистоту git-состояния спеки перед мержем — точка отката в `HEAD`, не в `.bak`.
- НИКОГДА не удаляй контент, не упомянутый в change (кроме явных REMOVED-секций).
- НИКОГДА не создавай `.bak`-файлы в репозитории.
- Проверяй семантический diff заголовков после мержа, а не только line count.
- Показывай что было сделано в итоге.
