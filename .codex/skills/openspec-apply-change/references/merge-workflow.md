# Merge-workflow: подробности Analyst Merge

Детальная механика мержа `change.md` в целевую спецификацию сервиса. Высокоуровневый workflow — в `SKILL.md`. Здесь — правила, таблицы, команды.

---

## 1. Точка отката — git, не `.bak`

Перед мержем проверь чистоту git-состояния целевой спеки:

```bash
git status --porcelain <spec-path>
```

- Файл имеет незакоммиченные изменения → попроси пользователя сначала закоммитить или откатить их. Иначе мерж смешает их с результатом apply, откат станет невозможен.
- Файл чист → продолжай. Текущий commit в `HEAD` и есть точка отката.

**Не создавай `.bak`-файлы.** Если пользователь хочет сравнить до/после — пусть использует `git diff <spec-path>` или `git show HEAD:<spec-path>`.

Откат после мержа: `git checkout HEAD -- <spec-path>`.

## 3. Перенумерация подразделов

После вставки новых подразделов **ОБЯЗАТЕЛЬНО** перенумеруй ВСЕ подразделы в затронутых секциях, чтобы нумерация была последовательной.

Пример: в секции 2.3 было 2.3.1–2.3.7, добавили 3 новых подраздела после 2.3.2. Итоговая нумерация должна быть 2.3.1–2.3.10. НЕ 2.3.1, 2.3.2, 2.3.8, 2.3.9, 2.3.10, 2.3.3, ….

Правила:

- Новые подразделы вставляются в логически правильное место (рядом с тематически близкими).
- Все последующие подразделы перенумеровываются.
- Все внутренние ссылки (`см. п. 2.3.2`) обновляются на новые номера.

---

## 4. Пересборка TOC

После мержа ОБЯЗАТЕЛЬНО обнови блок `<!-- TOC --> ... <!-- TOC -->` в начале файла.

Правила пересборки:

- Просканируй ВСЕ заголовки `##`, `###`, `####` в файле.
- Для каждого заголовка сгенерируй markdown-ссылку с якорем.
- Якорь: lowercase, пробелы → `-`, удаляются спецсимволы.
- Уровни вложенности: `##` без отступа, `###` с 2 пробелами, `####` с 4 пробелами.
- Сохраняй HTML-теги (напр. `<font>`) в тексте ссылки, но НЕ в якоре.

Формат строки TOC:

```
* [2.3.8. Правила классификации email bounce](#238-правила-классификации-email-bounce)
```

## 5. Версия спеки

**Инкрементируй версию** спецификации в шапке (напр. `1.0` → `1.1`).

---

## 6. Форматирование

При мерже **сохраняй HTML-разметку** в исходном тексте спеки:

- Цветовые маркеры: `<font color="red">**Alt**</font>`, `<font color="blue">**Opt**</font>`, `<font color="green">**Примечание:**</font>`.
- Прочие HTML-теги в тексте спецификации.

Не теряй их при вставке нового контента.

## 6.1. Перекрёстные ссылки и инварианты

При мерже разделов 3 (Глоссарий), 4 (Бизнес-инварианты), 5 (FSM) из change.md:

- Ссылки на правила из guard в FSM (`см. правило 2.X`) обновляй на актуальную нумерацию правил в спеке после перенумерации (§ 3).
- Если новый инвариант (раздел 4) имеет в change.md колонку «Покрывающий критерий приёмки» — проверь, что соответствующая строка добавлена в раздел «Рекомендации к критериям приёмки» (раздел 3 спеки) и ссылка валидна.
- Если новый переход FSM (раздел 5) имеет покрытие в критериях приёмки (раздел 16 change.md → раздел 3 спеки) — проверь ссылку на номер перехода `2.4.X`.
- Удалённые инварианты / переходы → найди в спеке все ссылки `см. инвариант 1.3.X` / `см. переход 2.4.X` на удалённые номера, отметь как «требуют ручной правки» в Output (не пытайся переписать автоматически).

## 7. Семантический diff заголовков (финальная проверка)

Команда через process substitution (bash в Linux/macOS):

```bash
diff \
  <(git show HEAD:<spec-path> | grep -E '^(##|###|####) ') \
  <(grep -E '^(##|###|####) ' <spec-path>)
```

Если shell не поддерживает process substitution (старый `sh`, Windows `cmd` без Git Bash) — используй временные файлы **в директории репо**, не в `/tmp`, и удали после проверки:

```bash
git show HEAD:<spec-path> | grep -E '^(##|###|####) ' > .headings-before.tmp
grep -E '^(##|###|####) ' <spec-path> > .headings-after.tmp
diff .headings-before.tmp .headings-after.tmp
rm -f .headings-before.tmp .headings-after.tmp
```

Убедись:

- **Ни один заголовок `##/###/####` из `HEAD` не исчез** (кроме явно указанных в REMOVED-секциях change.md). Переформулирование допустимо, исчезновение без REMOVED — запрещено.
- Нумерация подразделов последовательная (нет пропусков или дублей).
- TOC соответствует фактическим заголовкам.
- Все внутренние ссылки (`см. п. X.Y`) указывают на существующие номера.

## 8. Статус change.md после мержа

Обнови шапку change.md:

```
> **Статус**: Реализовано
```

Меняй ТОЛЬКО строку со статусом, остальное не трогай.

---

## 9. Что НЕ делать

- НЕ удаляй существующий контент, не упомянутый в change (кроме явных REMOVED-секций).
- НЕ создавай `.bak`-файлы.
- НЕ проверяй мерж по line count — только семантический diff заголовков (§ 7).
- НЕ привязывайся к номерам разделов — используй ТЕКУЩУЮ нумерацию и структуру целевой спецификации.
