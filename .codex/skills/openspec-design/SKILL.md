Создать технический проект реализации (design.md) и план задач (tasks.md) на основе change.md.

**Input**: Опционально — имя change. Если не указано, определить автоматически.

**Bundle-пути.**

- Собственные `templates/X` — относительно директории этого скила.
- Внешние файлы из `openspec-explore` — **сосед по директории**, резолв через `<Skill dir>/../openspec-explore/references/<file>`.
- Если harness даёт строку `Skill directory: <abs>` в обёртке активации — используй её как корень. Например: `Read("<Skill dir>/templates/design.md")`, `Read("<Skill dir>/../openspec-explore/references/code-analysis-priority.md")`.
- Если строки нет — один раз `Glob("**/openspec-skills/skills/openspec-design/templates/<file>")` и `Glob("**/openspec-skills/skills/openspec-explore/references/<file>")`, бери первый результат каждого.
- Пусто → спроси пользователя путь установки, не ищи вручную.

**Bundled materials** (относительно этого SKILL.md):

- `templates/design.md` — шаблон технического проекта (структура разделов, примеры).
- `templates/tasks.md` — шаблон плана задач.

**Внешние references (скил `openspec-explore`):**

- `openspec-explore/references/code-analysis-priority.md` — приоритет инструментов анализа кода (читай по мере необходимости).

**AskUserQuestion fallback**: если инструмент недоступен в клиенте — задай тот же вопрос обычным текстом и дождись ответа пользователя. Не угадывай.

## Steps

### 1. Выбрать change

Если имя указано — использовать его. Иначе:
```bash
ls openspec/changes/ 2>/dev/null
```
- Автовыбор, если только один active change (исключая `archive/`)
- Если несколько — предложить выбор

Объявить: "Проектирую change: <name>"

### 2. Проверь наличие change.md

Прочитай `openspec/changes/<name>/change.md`.

**Если файл не существует:** предложить сначала запустить скилл `openspec-propose`.

**Если design.md уже существует:** спросить — продолжить редактирование или создать заново.

### 3. Прочитай change.md

Изучи все 16 разделов change.md:
- Предложение (раздел 1)
- Бизнес-логика (раздел 2)
- Глоссарий (раздел 3)
- Бизнес-инварианты (раздел 4)
- Состояния и переходы FSM (раздел 5)
- Модели данных (раздел 6)
- Интеграции (раздел 7)
- Обработка ошибок (раздел 8)
- Хедеры и метаданные (раздел 9)
- Валидация (раздел 10)
- Безопасность (раздел 11)
- Миграция (раздел 12)
- Логирование (раздел 13)
- Мониторинг (раздел 14)
- Конфигурация (раздел 15)
- Критерии приёмки (раздел 16)

### 4. Изучи контекст

a. **Проверь наличие `.research-notes.md` рядом с change.md** — `openspec/changes/<name>/.research-notes.md`. Если файл существует и актуален (создан на шаге `openspec-propose`) — используй агрегат оттуда как источник информации о текущей реализации. Это экономит время: исследование уже проведено.

b. Прочитай целевую спецификацию — путь в шапке change.md, поле «Целевая спецификация».

c. Прочитай build-файлы / манифесты зависимостей для понимания стека: `pom.xml`, `build.gradle`, `go.mod`, `package.json`, `pyproject.toml`, `requirements.txt`, `Cargo.toml` — что есть в проекте.

d. Для design-решений делай **точечные reads** — не более 3–5 файлов за раз, только непосредственно затронутые узлы (классы/модули/функции, напрямую упомянутые в change.md, плюс их ближайшие вызывающие слои). Запрещено: массовое сканирование каталогов, рекурсивный обход всех связанных сущностей — этим занимается `openspec-explore`. Если нужно шире — прерви шаг и запусти `openspec-explore` отдельно.

e. **Анализ кода** — приоритет: Serena (MCP) → LSP → embeddings → прочие MCP → Grep/Glob. Подробнее — `openspec-explore/references/code-analysis-priority.md`.

Если `.research-notes.md` отсутствует и change нетривиален — предложи пользователю вернуться в `openspec-propose` для полного исследования либо продолжи в ручном режиме, явно обозначив риск неполноты.

### 5. Создай директорию design-артефактов

```bash
mkdir -p openspec/changes/<name>
```

### 6. Прочитай шаблоны

Шаблоны поставляются в составе скила — `templates/design.md`, `templates/tasks.md`.

### 7. Создай design.md

Заполни разделы шаблона `templates/design.md` реальными данными из change.md и изученного кода. Структура разделов, описания, примеры — в шаблоне; не перечисляй их заново.

**Ссылайся на конкретные сущности кода проекта** (классы, модули, пакеты, функции — в терминах стека). Не проектируй вслепую.

**Уровни обязательности требований:**
- ДОЛЖЕН / ОБЯЗАН — обязательно для реализации
- СЛЕДУЕТ — рекомендуется
- МОЖЕТ — опционально

Удали все HTML-комментарии из шаблона. Запиши в `openspec/changes/<name>/design.md`.

### 8. Создай tasks.md

На основе design.md и критериев приёмки из change.md:

- Группируй задачи логически (подготовка, core, тестирование, финализация)
- Каждая задача — чекбокс формат: `- [ ] X.Y Описание`
- Порядок задач — по зависимостям
- Задачи должны быть достаточно маленькими для выполнения за одну сессию

Включи чеклист верификации (КОГДА/ТОГДА из change.md раздел 16 «Рекомендации к критериям приёмки»).

Запиши в: `openspec/changes/<name>/tasks.md`

### 9. Предложи ревью

После создания предложи пользователю проверить и дать обратную связь.

## Guardrails

- ВСЕГДА читай change.md перед началом
- ВСЕГДА изучай существующий код (не проектируй вслепую)
- НЕ придумывай технологии/библиотеки которых нет в проекте
- Если контекст неясен — задавай вопросы, не делай предположений
- design.md ДОЛЖЕН ссылаться на конкретные сущности из кода проекта (классы/модули/пакеты/функции — по стеку)
- tasks.md должен быть реализуемым — не добавляй нереальные задачи
- При обновлении файлов сохраняй уже заполненные данные если они корректны

## Обновление при обратной связи

Если пользователь даёт обратную связь:
1. Прочитай текущие design.md и/или tasks.md
2. Примени запрошенные изменения
3. Сохрани файл
4. Сообщи что изменено
