# Section Guide — заполнение service-spec.md

Как правильно заполнять разделы спецификации.

## Общие правила

- Удали ВСЕ HTML-комментарии (`<!-- ... -->`) из результата.
- Заполняй КАЖДЫЙ раздел реальным содержимым, без плейсхолдеров.
- Если раздел не применим (например, мониторинг для прототипа) — напиши: *"Не определено на текущем этапе. Будет добавлено при подготовке к production."*
- Записывай файл в `openspec/specs/<service-name>/<service-name>.md`.

## Уровни обязательности требований

- **ДОЛЖЕН / ОБЯЗАН** — обязательно, система не работает без этого.
- **СЛЕДУЕТ** — рекомендуется, допускается отступление с обоснованием.
- **МОЖЕТ** — опционально.

## Форматы ключевых блоков

- **Глоссарий (1.2)** — таблица «Термин / Синонимы / Определение». Каждый доменный термин, появляющийся в правилах, моделях, сценариях, обязан быть здесь.
- **Бизнес-инварианты (1.3)** — нумерованный список с формулировкой **ВСЕГДА** / **НИКОГДА**. Каждый инвариант наблюдаем (можно проверить тестом или запросом). Покрываются критериями приёмки в разделе 3.
- **Типовой флоу клиента (2.1.1)** — таблица «# / Шаг клиента / Endpoint / Зачем / Что клиент получил». Последовательность API-вызовов клиента от начала пользовательского сценария до завершения. Если сценариев несколько (покупатель/админ/оператор) — отдельная таблица на каждый. Если сервис чисто событийный — писать «Не применимо: сервис не имеет клиентского API».
- **Обработчики по триггерам (2.1.2.N)** — один подраздел на каждый триггер (HTTP endpoint / Kafka-consumer / cron / gRPC / SSE / webhook). Используется ФИКСИРОВАННЫЙ скелет: **Триггер, Инициатор, Вход, Предусловия (guard), Шаги обработки (нумерованные, со ссылками на 2.3 и 2.4), Side-effects, Выход/ответ, Ошибочные ветки (ссылка на 2.2 и 2.11), Идемпотентность (ключ и поведение при повторе)**. Пропуск любого поля скелета — замечание ревью. Это ядро «восстановимости» сервиса по спеке.
- **Бизнес-правила (2.3)** — каждое правило состоит из ДВУХ частей: (1) ФОРМУЛИРОВКА (ЕСЛИ/ТОГДА/ИНАЧЕ и/или математика); (2) ПОШАГОВЫЙ БИЗНЕС-ПСЕВДОКОД — нумерованные шаги с явной обработкой краёв (null, пустой ввод, округления, гонки). Псевдокод ОБЯЗАТЕЛЕН для КАЖДОГО правила (даже линейного). Язык — независимый от языка реализации (без Java/Python/Go-синтаксиса, без аннотаций фреймворков). Правила перехода статусов сюда НЕ включаются — они в 2.4 FSM.
- **FSM (2.4)** — таблица переходов (#, from, to, trigger, guard, side-effect) + список состояний с семантикой + явные запрещённые переходы + диаграмма (PlantUML/Mermaid). Если у сущности нет состояний — пишется «Не применимо для данного сервиса». Guard-условие в FSM ссылается на правило в 2.3, а не дублирует его.
- **Критерии приёмки (3)** — КОГДА / ТОГДА (примеры в шаблоне). Каждый переход FSM из 2.4 и каждый бизнес-инвариант из 1.3 ДОЛЖНЫ иметь хотя бы один критерий приёмки.
- **API** — метод, путь, request/response JSON-схема.
- **Модели данных** — таблица полей (имя, тип, обязательность, пример, описание).
- **Интеграции** — протокол, формат, таймаут, retry-политика.
- **Конфигурация** — таблица параметров (имя, тип, default, описание).

## Перекрёстные ссылки (ОБЯЗАТЕЛЬНО)

Спецификация должна быть прошита перекрёстными ссылками — это и есть главный механизм «восстановимости» сервиса по документу:

- Шаги клиентского флоу 2.1.1 со значимым переходом состояния → ссылка на переход **2.4.X**.
- Шаги обработчика 2.1.2.N со значимым решением → ссылка на правило **2.3.X** и/или переход **2.4.X**.
- Поле «Предусловия» обработчика 2.1.2.N → ссылка на инвариант **1.3.X** и/или переход **2.4.X**.
- Поле «Ошибочные ветки» обработчика 2.1.2.N → ссылка на сценарий **2.2.X** и/или код ошибки **2.11**.
- Альтернативные сценарии 2.2 → ссылка на правило **2.3.X** и/или переход **2.4.X**.
- Поля модели данных 2.7 со значениями статуса/типа → ссылка на 2.4 (для статусов) или на глоссарий 1.2 (для доменных типов).
- Конфигурационные параметры 2.8, влияющие на поведение, → ссылка на правило 2.3.X, переход 2.4.X, или обработчик 2.1.2.N.
- Критерии приёмки 3 → ссылка на переход FSM (2.4.X) или инвариант (1.3.X), который проверяет.
- Параметры аудита/безопасности 4.1, 4.5 → ссылка на инвариант 1.3.X.

Допустимый формат ссылки в тексте: `см. правило 2.3.X`, `см. переход 2.4.X`, `см. инвариант 1.3.X`, `см. глоссарий 1.2`, `см. обработчик 2.1.2.N`, `см. сценарий 2.2.X`.

## Большие файлы

Спецификация может получиться 300+ строк. Если файл большой — записывай его **по частям** (раздел за разделом), чтобы не потерять контент из-за ограничений контекстного окна. Предупреди пользователя: *"Спецификация объёмная, записываю по частям."*

## Ревью после создания

Проверь:

1. **Самодостаточность** — можно ли по одной спеке восстановить сервис с нуля, не имея доступа к исходному коду? Каждый раздел должен быть полным описанием, а не отсылкой к коду.
2. **Отсутствие утечек реализации** — нет ли ссылок на классы (`OrderService`), методы (`createOrder()`), пакеты (`com.shop.orders`), файлы исходного кода (`src/main/java/...`)? Допустимы только ссылки на контрактные артефакты (OpenAPI/proto/avro/JSON Schema/SQL DDL/диаграммы).
3. **Полнота** — все ли функции сервиса отражены? Нет ли пропущенных эндпоинтов, сущностей, интеграций, бизнес-правил?
4. **Точность** — соответствуют ли описания наблюдаемому поведению сервиса? Нет ли расхождений в названиях полей API, типах, форматах данных, кодах ошибок?
5. **Непротиворечивость** — нет ли конфликтов между разделами?
6. **Конкретность** — нет ли размытых формулировок, плейсхолдеров, TODO?
7. **Структура** — все ли обязательные разделы присутствуют и заполнены?
8. **Глоссарий** — каждый доменный термин из правил/моделей/сценариев определён в 1.2? Нет ли терминов, использующихся в разном смысле в разных разделах?
9. **Инварианты** — раздел 1.3 содержит нерушимые гарантии в формате ВСЕГДА/НИКОГДА? Каждый инвариант покрыт критерием приёмки в разделе 3?
10. **Типовой флоу клиента (2.1.1)** — присутствует таблица последовательности API-вызовов клиента? Для чисто событийных сервисов — явная пометка «Не применимо»? Шаги ссылаются на переходы 2.4.X при смене статуса сущности?
11. **Обработчики (2.1.2.N)** — на КАЖДЫЙ триггер сервиса (HTTP endpoint, Kafka-consumer, cron, gRPC, SSE, webhook) есть отдельный подраздел? В каждом заполнены ВСЕ поля скелета (Триггер, Инициатор, Вход, Предусловия, Шаги, Side-effects, Выход/ответ, Ошибочные ветки, Идемпотентность)? Нет ли триггеров, упомянутых в 2.5/2.6, но не имеющих обработчика в 2.1.2?
12. **Бизнес-правила (2.3)** — КАЖДОЕ правило содержит и ФОРМУЛИРОВКУ (ЕСЛИ/ТОГДА), и ПОШАГОВЫЙ ПСЕВДОКОД (нумерованные шаги)? Псевдокод без привязки к языку реализации? Края (null, округление, пустой ввод, гонки) явно обработаны?
13. **FSM** — если сервис управляет жизненным циклом сущности, раздел 2.4 содержит полную таблицу переходов и список запрещённых переходов? Нет ли неявных переходов, упомянутых в сценариях, но отсутствующих в FSM?
14. **Перекрёстные ссылки** — шаги обработчиков 2.1.2.N ссылаются на правила 2.3.X и переходы 2.4.X? Поле «Ошибочные ветки» ссылается на 2.2.X и 2.11? Критерии приёмки 3 ссылаются на FSM/инварианты?

Если доступен tool запуска субагента — запусти **отдельный субагент** с промптом:
> Проведи ревью файла `openspec/specs/<service-name>/<service-name>.md`. Ключевой критерий — **самодостаточность**: может ли разработчик реализовать сервис с нуля по одной этой спеке, не заглядывая в исходный код? Проверь дополнительно: (а) отсутствие утечек реализации (имена классов/методов/пакетов/файлов исходного кода недопустимы; ссылки на контракты — OpenAPI/proto/avro/SQL DDL/диаграммы — допустимы); (б) глоссарий 1.2 — каждый доменный термин определён; (в) инварианты 1.3 — нерушимые правила ВСЕГДА/НИКОГДА, каждый покрыт критерием приёмки; (г) 2.1.1 Типовой флоу клиента — таблица последовательности API-вызовов присутствует, либо явная пометка «Не применимо»; (д) 2.1.2.N Обработчики — на каждый триггер (endpoint/consumer/cron/gRPC/SSE/webhook) отдельный подраздел со ВСЕМИ полями скелета (Триггер, Инициатор, Вход, Предусловия, Шаги, Side-effects, Выход/ответ, Ошибочные ветки, Идемпотентность); (е) 2.3 Бизнес-правила — каждое правило имеет и формулировку, и пошаговый псевдокод без привязки к языку реализации; края (null, округление, гонки) явно обработаны; (ж) FSM 2.4 — полная таблица переходов и запрещённых переходов; (з) перекрёстные ссылки обработчик↔правило↔FSM↔инвариант↔критерий приёмки; (и) полноту, точность, непротиворечивость, конкретность, структуру. Верни список проблем или "Ревью пройдено, замечаний нет".

Если tool запуска субагента недоступен — проведи ревью самостоятельно по этому чеклисту.

Если найдены проблемы — исправь спецификацию и сообщи пользователю, что было исправлено.

## Output

После создания и ревью покажи:
- Название сервиса и путь к файлу.
- Список сгенерированных разделов.
- Ключевые задокументированные интеграции.
- Результат ревью (замечания и исправления, если были).
- Подсказку: *"Спецификация готова. Для внесения изменений запусти скилл `openspec-propose`."*
