# /sdd:implement

Реализуй конкретную задачу из плана. Пиши тесты перед кодом (test-first).

## Входные данные

Задача из задач: `{{args}}` или путь к спецификации и номер задачи (TSK-N)

## Что читать

1. `docs/sdd/constitution.md` — конституционные ограничения (обязательно!)
2. `docs/specs/000N-slug/spec.md` — требования и acceptance criteria
3. `docs/specs/000N-slug/plan.md` — технический подход
4. `docs/specs/000N-slug/tasks.md` — текущая задача и её вход/выход
5. `docs/specs/000N-slug/test-plan.md` — что тестировать
6. Существующий код — изучить перед изменениями

## Что делать

1. Определить текущую задачу из tasks.md (первую pending)
2. Написать/обновить тесты ДО реализации (test-first)
3. Реализовать минимальные изменения, необходимые для задачи
4. Запустить команды верификации:
   - `mvn compile` — компиляция без ошибок
   - `mvn test` — все тесты проходят
5. Обновить `task-state.md` и `work-log.md`
6. Заполнить матрицу трассировки в spec.md

## Разрешено изменять

- `src/` — код приложения
- `tests/` — тесты
- `docs/specs/000N-slug/task-state.md` — статус задач
- `docs/specs/000N-slug/work-log.md` — лог работы
- `docs/specs/000N-slug/spec.md` — матрица трассировки (только статус)

## ЗАПРЕЩЕНО изменять

- `docs/sdd/` — governance docs
- ADR без явного запроса
- Другие спеки
- Секреты, токены, пароли

## Gate

Задача завершена, когда:
- Все acceptance criteria для REQ выполнены
- `mvn test` проходит без регрессий
- `task-state.md` обновлён
- Отчёт предоставлен: изменённые файлы, команды, spec compliance, risks
