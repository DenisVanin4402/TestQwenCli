# /sdd:plan

Создай технический план реализации на основе утверждённой спецификации.

## Входные данные

Путь к спецификации или `{{args}}`

## Что читать

1. `docs/specs/000N-slug/spec.md` — утверждённая спецификация
2. `docs/sdd/constitution.md` — конституционные ограничения
3. `docs/sdd/gates.md` — Plan Ready gate
4. `docs/specs/_template/plan.md` — шаблон плана
5. Существующий код — изучить текущую архитектуру (grep, glob, read_file)
6. Существующие ADR в `docs/adr/` — проверить применимость

## Что делать

1. Определить затронутые компоненты и файлы
2. Выбрать технический подход с обоснованием
3. Описать альтернативы и почему они не выбраны
4. Определить зависимости (внешние и внутренние)
5. Выявить изменения контрактов (API, events)
6. Определить технические риски
7. Создать `docs/specs/000N-slug/plan.md`
8. Если есть значимое архитектурное решение — предложить создание ADR

## Разрешено изменять

- `docs/specs/000N-slug/plan.md` — создание плана
- `docs/specs/000N-slug/research.md` — если нужен анализ альтернатив

## ЗАПРЕЩЕНО изменять

- Код
- Существующие ADR без явного запроса

## Gate

Остановиться после создания plan.md. Предложить следующий шаг:
- Если есть архитектурное решение → `/sdd:plan` с ADR
- Иначе → `/sdd:tasks`
