# SDD Diff Analyzer

Навык для анализа git diff, классификации изменений, построения карты влияния и генерации контекстного пакета для delta-specification.

## Входные данные

- Две git-ссылки для сравнения (ветки, коммиты, теги)
- Опционально: MR/PR описание или URL

## Процедура

### 1. Извлечение diff

Запустить diff-extractor:

```bash
# Linux/macOS
bash scripts/sdd-diff-extract.sh <ref1> <ref2> [target-dir]

# Windows
scripts\sdd-diff-extract.bat <ref1> <ref2> [target-dir]
```

Результат: `diff-summary.md` и `impact-map.md`.

### 2. Классификация изменений

Каждый изменённый файл классифицируется по одному из четырёх типов:

| Тип | Паттерны файлов | Примеры |
|-----|----------------|---------|
| **Контракты (API/DBML)** | `openapi.yaml`, `asyncapi.yaml`, `*.dbml` | API-спецификации, схемы БД |
| **Требования (.md)** | `docs/*.md`, `*.md` | Бизнес-требования, правила |
| **Код (src/)** | `src/*.java`, `src/*.py`, и т.д. | Исходный код приложения |
| **Конфигурации** | `*.properties`, `*.yml`, `*.yaml`, `pom.xml` | Настройки приложения |

Для каждого файла фиксируется:
- Тип изменения: Added / Modified / Deleted
- Количество добавленных/удалённых строк
- Классификация (тип из таблицы выше)

### 3. Построение карты влияния (Impact Map)

**Прямое влияние:**
- Все файлы, непосредственно затронутые в diff
- Сгруппированные по директориям

**Косвенное влияние:**
- Если изменены `openapi.yaml` → проверить AGENTS.md, существующие спеки с API-требованиями, ADR об API, тесты эндпоинтов
- Если изменены `.md` файлы требований → проверить constitution.md, связанные спеки, AGENTS.md
- Если изменён код в `src/` → проверить тесты, contract tests, связанные спеки
- Если изменены конфигурации → проверить совместимость настроек, CI/CD

**Затронутые спеки:**
- Поиск по совпадению имён/ключевых слов с существующими спеками в `docs/specs/`

**Затронутые ADR:**
- Поиск по упоминаниям ADR в изменённых файлах

### 4. Генерация контекстного пакета

На основе diff-summary и impact-map формируется контекстный пакет:

```
# Context Packet: Delta-Analysis

## Цель
Проанализировать изменения из MR и подготовить delta-spec

## Источник
- Ветки: <ref1>..
- Diff: diff-summary.md
- Impact: impact-map.md

## Классификация изменений
| Тип | Кол-во файлов | Ключевые файлы |
| Контракты | N | ... |
| Требования | N | ... |
| Код | N | ... |
| Конфиги | N | ... |

## Затронутые области
- Прямое: список файлов
- Косвенное: рекомендуемые проверки

## Рекомендуемый context packet
Файлы, которые следует прочитать для генерации delta-spec:
1. Изменённые .md файлы (для понимания бизнес-логики)
2. Изменённые контракты openapi/asyncapi
3. Существующие спеки, связанные по именам
4. Затронутые ADR
```

### 5. Определение требований из diff

Автоматически выявить требования на основе изменений:

- **Изменения .md файлов →** требования к бизнес-логике
- **Изменения openapi.yaml →** требования к API-контрактам
- **Изменения кода src/ →** требования к реализации (уже частично реализованы)
- **Изменения конфигураций →** нефункциональные требования

Каждое выявленное требование получает ID:
- Новые: `REQ-NEW-1`, `REQ-NEW-2`, ...
- Изменённые существующие: с пометкой «изменено в части ...»

## Результат

- `diff-summary.md` — структурированное резюме изменений
- `impact-map.md` — карта влияния
- Контекстный пакет для генерации delta-spec

## Когда использовать

- Аналитик открыл MR с бизнес-изменениями
- Нужно быстро понять что изменилось без чтения всех файлов
- Подготовить delta-spec для последующей реализации

## Когда НЕ использовать

- Если MR содержит только технические изменения (refactor, formatting) без бизнес-логики
- Если delta-spec уже существует для данного MR
- Если изменения незначительны (менее 3 файлов, только комментарии/whitespace)

## Связанные артефакты

- Команда: `/sdd:from-diff`
- Шаблон: `docs/specs/_template/delta-spec.md`
- Скрипты: `scripts/sdd-diff-extract.sh`, `scripts/sdd-diff-extract.bat`
