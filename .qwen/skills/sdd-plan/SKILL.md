# SDD Plan

## Входные данные

- Утверждённая спецификация `docs/specs/<id>-<slug>/spec.md`
- Существующий код (через exploration)

## Процедура

1. Прочитать spec.md — понять требования и acceptance criteria.
2. Изучить существующий код — определить затронутые файлы.
3. Выбрать технический подход с обоснованием.
4. Описать альтернативы (minimum 2) и почему они не выбраны.
5. Определить зависимости и риски.
6. Выявить контракты, которые нужно обновить.
7. Определить, нужна ли ADR (архитектурное решение с последствиями).
8. Создать `plan.md` по шаблону `docs/specs/_template/plan.md`.
9. В конце `plan.md` дописать фразу «Конец файла» — это важно для последующей корректной обработки.

## Результат

- `docs/specs/<id>-<slug>/plan.md` заполнен
- Если есть ADR — `docs/adr/<id>-<title>.md`

## Gate

Остановиться после создания plan.md. Предложить следующий шаг:
- Если есть значимое архитектурное решение → создать ADR
- Иначе → перейти к `/sdd:tasks`

## Когда НЕ использовать

- Если plan уже существует и не изменился scope.
- Если задача тривиальная (менее 1 file change).
