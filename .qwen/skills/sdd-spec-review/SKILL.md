# SDD Spec Review

## Входные данные

- Запрос пользователя или issue
- Соответствующие `docs/specs/**`
- Текущий diff (если есть)

## Процедура

1. Определить релевантную спецификацию и requirement IDs.
2. Сравнить запрошенное поведение с существующими acceptance criteria.
3. Перечислить неоднозначности как `[NEEDS CLARIFICATION]`.
4. Определить какие тесты/контракты должны существовать или быть обновлены.
5. Не редактировать файлы, если явно не запрошено.

## Что проверять

| Проверка | Что искать |
|----------|------------|
| Completeness | Все REQ покрыты AC? |
| Ambiguity | Неоднозначные формулировки в требованиях |
| Alignment | Запрос соответствует spec? |
| Contracts | API-контракты обновлены? |
| NFR | Security/privacy/performance учтены? |

## Результат

- Релевантные требования
- Отсутствующие/неоднозначные требования
- Обязательные тесты/контракты
- Риски и рекомендуемый следующий гейт
