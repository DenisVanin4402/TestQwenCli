# Repository Guidelines

## Структура проекта и модулей

Это multi-module Maven-проект на Java 21 и Spring Boot 3.4.3. Корневой `pom.xml` агрегирует модули:

- `test-qwen-cli-app`: основное Spring Boot-приложение, gateway-контроллеры, сервисы, репозитории, конфигурация и интеграция dashboard.
- `dashboard-backend`: backend-модель и REST API для тестового dashboard.
- `dashboard-ui`: статические SPA-ресурсы dashboard.

Код Java находится в `*/src/main/java`, тесты - в `*/src/test/java`. Конфигурация приложения хранится в `test-qwen-cli-app/src/main/resources`. OpenAPI-контракты лежат в `docs/openapi`, SDD/OpenSpec-материалы - в `docs/sdd` и `openspec`. Каталог `target/` создается Maven и не редактируется вручную.

## Базовая инструментальная гигиена

- Для поиска по репозиторию сначала используйте `rg` или `rg --files`. Если `rg` недоступен, используйте ближайший быстрый аналог и явно укажите это в отчете о работе.
- Перед чтением больших деревьев сужайте scope: `-g`, конкретный модуль, каталог, расширение файла, исключение `target/`.
- Для ручных правок используйте единый patch-инструмент. Не переписывайте исходники и документацию через `cat`, PowerShell redirection, heredoc/herestring или самописные rewrite-скрипты, если достаточно точечного патча.
- Форматтеры и генераторы могут менять файлы автоматически, но их запуск должен быть отдельным осознанным шагом с последующей проверкой diff.
- Перед изменением файла читайте его актуальное содержимое: рабочее дерево может содержать незакоммиченные пользовательские правки.

## Команды сборки, тестов и запуска

- `mvn test`: запускает все тесты JUnit/Spring во всех модулях.
- `mvn -pl test-qwen-cli-app test`: запускает тесты только основного приложения.
- `mvn package`: компилирует проект, запускает тесты и собирает артефакты в `target/`.
- `mvn spotless:check`: проверяет базовое форматирование файлов, настроенное в корневом `pom.xml`.
- `mvn spotless:apply`: применяет базовое форматирование. После запуска обязательно проверьте diff.
- `mvn checkstyle:check`: запускает Java-linting по `config/checkstyle/checkstyle.xml`.
- `mvn -pl test-qwen-cli-app spring-boot:run`: запускает основное приложение локально. Корневой эндпоинт должен вернуть `TestQwenCli is running` по адресу `http://localhost:8080/`.

Команды выполняются из корня репозитория. При изменении зависимостей обновляйте нужный `pom.xml` и повторно запускайте релевантные `mvn test`, `mvn spotless:check` и `mvn checkstyle:check`.

## Стиль кода и соглашения об именах

Следуйте текущему стилю Spring Boot: пакеты пишутся в нижнем регистре, классы в `PascalCase`, методы и поля в `camelCase`. Имена тестов должны описывать проверяемое поведение, например `rootEndpointReturnsStatusMessage`.

Документация проекта, Markdown-файлы, JavaDoc, комментарии в коде, пояснения в конфигурации и новые сопроводительные материалы должны быть на русском языке. Имена классов, методов, пакетов, технические команды и внешние протокольные значения остаются на английском.

Используйте возможности Java 21 умеренно. Держите импорты чистыми, не добавляйте неиспользуемые зависимости и сохраняйте стиль отступов соседних файлов.

## Правила тестирования

Добавляйте или обновляйте тесты для каждого изменения поведения. Для веб-эндпоинтов используйте Spring Boot Test и MockMvc. Размещайте тесты в соответствующем пакете внутри `src/test/java` и предпочитайте названия, описывающие поведение, а не детали реализации.

Перед отправкой изменений запускайте минимум релевантный модульный тест. Для изменений общего поведения запускайте `mvn test`. Для PR-ready проверки используйте `mvn test`, `mvn spotless:check` и `mvn checkstyle:check`. Для эндпоинтов по возможности проверяйте и HTTP-статус, и содержимое ответа.

## Запреты на destructive actions

Без явного подтверждения пользователя нельзя выполнять:

- удаление или рекурсивное перемещение файлов и каталогов: `rm`, `Remove-Item`, `del`, `rmdir`, `Move-Item` с широким scope;
- сброс или массовую очистку Git-состояния: `git reset --hard`, `git clean`, `git checkout --`, force push;
- destructive DB-команды: `DROP`, `TRUNCATE`, массовые `DELETE`/`UPDATE`, миграции против неявного окружения;
- удаление Docker volumes/images/containers, если это может уничтожить состояние;
- deploy, restart production-like сервисов, изменение секретов или прав доступа.

Перед потенциально destructive действием сначала покажите точный target, причину, ожидаемый эффект и безопасный dry-run, если инструмент его поддерживает.

## Коммиты и pull request

Текущая история Git использует короткие сообщения в нижнем регистре, например `add research and plan for sdd`. Пишите коммиты кратко и в повелительной форме: `add gateway protocol docs`, `test root endpoint response`.

Pull request должен содержать краткое описание, результат проверки, например `mvn test`, и ссылки на связанные задачи или документы. Скриншоты или примеры ответов добавляйте только при изменении пользовательского HTTP-поведения.

## Безопасность и конфигурация

Не коммитьте секреты, локальные учетные данные и пути, зависящие от машины разработчика. Значения для окружений выносите во внешнюю конфигурацию, а обязательные свойства документируйте в `docs/` или комментариях `application.properties`.
