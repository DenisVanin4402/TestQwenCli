# Соглашения проекта

Этот файл фиксирует переносимые правила, которые дополняют `AGENTS.md`, `QWEN.md` и SDD-артефакты. Если правила расходятся, источник истины выбирается в таком порядке: явный запрос пользователя, `docs/specs/**` для текущей задачи, `docs/sdd/**`, затем `AGENTS.md`, `QWEN.md` и этот файл.

## Локальные команды

Все команды выполняются из корня репозитория.

```bash
mvn test
mvn spring-boot:run
mvn package
```

## Структура SDD

Существенная задача должна иметь каталог `docs/specs/<id>-<slug>/`.

Поддерживаются два режима:

- `spec-first`: первая спецификация создается из бизнес-запроса.
- `diff-driven`: первая спецификация синтезируется из бизнес-требования, analyst MR diff, измененной документации, OpenAPI, DBML и минимального source context.

Минимальный набор артефактов:

- `spec.md` - проблема, пользователи, цели, non-goals, требования и критерии приемки.
- `plan.md` - технический подход, затронутые файлы, контракты, риски.
- `tasks.md` - проверяемые задачи с requirement IDs и ownership scope.
- `test-plan.md` - стратегия проверки, команды и матрица трассировки.
- `work-log.md` - компактный журнал фактов, проверок и решений.
- `handoff.md` - контракт передачи, если работа не завершена в одном проходе.

Для `diff-driven` режима дополнительно используются:

- `intake.md` - бизнес-вход, MR, base branch и analyst branch.
- `diff-map.md` - карта измененных файлов и semantic blocks.
- `impact-map.md` - direct и indirect impacts.
- `source-context.md` - минимальный контекст с source references.
- `contracts/openapi-diff.md` - изменения API.
- `contracts/dbml-diff.md` - изменения модели данных.

## Идентификаторы

- Требования именуются `REQ-<номер>`.
- Критерии приемки именуются `AC-<номер>`.
- Задачи именуются `TASK-<номер>`.
- Архитектурные решения именуются `ADR-<номер>`.

Пример ссылки: `TASK-2` реализует `REQ-3` и проверяется `AC-4`.

## Когда нужен ADR

ADR требуется, если изменение:

- меняет публичный контракт API или формат данных;
- добавляет production-зависимость;
- меняет архитектурную границу, модель хранения или стратегию безопасности;
- вводит решение, которое будет трудно откатить;
- фиксирует значимый trade-off между несколькими вариантами.

ADR не нужен для локального рефакторинга, исправления теста или малой реализации, если архитектурный выбор уже очевиден из существующего кода.

## Definition of Ready

Задача готова к реализации, если:

- есть `spec.md` и `plan.md`;
- критические требования не содержат незакрытых уточнений;
- требования имеют ID и проверяемые acceptance criteria;
- понятны затронутые файлы, контракты и ограничения;
- в `tasks.md` задачи связаны с requirement IDs;
- определены команды проверки.

Для `diff-driven` задачи дополнительно:

- diff воспроизводим;
- измененные файлы классифицированы;
- каждый requirement имеет source reference;
- OpenAPI и DBML изменения вынесены в contract/data diff artifacts.

## Definition of Done

Задача завершена, если:

- код, тесты и документация соответствуют `docs/specs/<id>-<slug>/`;
- релевантные проверки запущены, а результат записан в `work-log.md`;
- матрица трассировки не содержит критических пробелов;
- ADR обновлен или создан, если было значимое решение;
- `handoff.md` заполнен, если остаются незакрытые действия.
