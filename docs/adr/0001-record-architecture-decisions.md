# ADR-0001: Использование SDD для агентской разработки

**Дата:** 2026-05-17
**Статус:** Принято
**Авторы:** Проектная команда

## Контекст

Проект TestQwenCli используется как полигон для Specification-Driven Development с AI coding-агентами. Без формализованного SDD-процесса агенты работают через историю чата, что приводит к:
- Потере контекста между сессиями
- Невозможности независимой верификации
- Дрейфу требований без traceability
- Повторению одних и тех же ошибок

## Решение

Принять Specification-Driven Development как основной процесс разработки в проекте, используя:
- Версионируемые Markdown-артефакты (`docs/specs/`)
- Qwen CLI команды и навыки для workflow
- Context packet архитектуру вместо full-repository context
- Human gates на спецификациях, ADR и приёмке
- Матрицу трассировки REQ -> AC -> Tests -> Code
- Независимый review через subagent

## Альтернативы

### 1. Vibe coding / ad-hoc prompting
- **Плюсы:** Быстрее на старте, меньше документации
- **Минусы:** Потеря контекста, дрейф требований, нет traceability, сложно верифицировать

### 2. Полная автоматизация через CI-пайплайн
- **Плюсы:** Автоматические проверки, consistent gates
- **Минус:** Значительная инфраструктура, не подходит для начального bootstrap

## Следствия

- **Положительные:**
  - Каждая задача имеет source-of-truth в спецификации
  - Агенты получают структурированный context packet
  - Верификация через матрицу трассировки
  - Human gate на критических решениях

- **Отрицательные:**
  - Дополнительные Markdown-файлы для каждой задачи
  - Требуется дисциплина от разработчиков и агентов
  - Spec lint требует поддержки
