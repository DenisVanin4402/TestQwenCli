# ADR-0001: Record Architecture Decisions

## Status

Accepted

## Context

SDD-процесс требует отделять требования, планы, задачи и устойчивые архитектурные решения. Без ADR значимые trade-offs остаются в чате или в `plan.md`, где их трудно найти после завершения задачи.

## Decision

Хранить архитектурные решения в `docs/adr/` как отдельные Markdown-файлы с монотонной нумерацией `ADR-0001`, `ADR-0002` и так далее.

ADR создается, когда изменение:

- меняет публичный API, контракт или формат данных;
- добавляет production-зависимость;
- меняет границу модулей, storage, security model или deployment-подход;
- фиксирует решение, которое сложно откатить;
- выбирает один вариант из нескольких значимых альтернатив.

## Alternatives

| Вариант | Плюсы | Минусы | Причина решения |
|---|---|---|---|
| Хранить решения только в `plan.md` | Меньше файлов | Решения теряются после закрытия задачи | Отклонено |
| Один общий decision log | Просто искать | Быстро превращается в append-only noise | Отклонено |
| Отдельные ADR-файлы | Хорошая трассировка и история | Нужно поддерживать нумерацию | Принято |

## Consequences

- Значимые решения можно ревьюить отдельно от реализации.
- Требуется обновлять ADR, если решение меняется.
- Мелкие локальные правки не требуют ADR.
