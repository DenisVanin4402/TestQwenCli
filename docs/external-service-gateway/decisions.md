# Архитектурные решения

## ADR-001. Выделяем отдельный сервис-шлюз

Решение: интеграция с внешним сервисом выносится в `external-service-gateway`.

Причины:

- внешний лимит `5` одновременных вызовов общий для нескольких доменных сервисов;
- `invest-pay` и `user-expertise` не имеют общей схемы данных;
- общая таблица-лимитер в одной из доменных БД создала бы скрытую связанность;
- шлюз централизует повторы, аудит, метрики и политику приоритета.

Последствия:

- появляется дополнительный внутренний сетевой переход;
- шлюз становится критическим техническим сервисом;
- зато доменные сервисы не знают о таблицах лимитирования и очередях.

## ADR-002. PostgreSQL используется как координатор v1

Решение: для v1 используем PostgreSQL сервиса-шлюза для lease-слотов и async-очереди.

Причины:

- ожидаемая нагрузка небольшая: около 1500 запросов/день, пики 10-20/мин;
- нужны транзакции, идемпотентность и постоянная очередь;
- `FOR UPDATE SKIP LOCKED` подходит для конкурентного захвата задач;
- `LISTEN/NOTIFY` уменьшает задержку без отдельного брокера.

Последствия:

- нужно следить за autovacuum и очисткой старых задач;
- PostgreSQL не является брокером сообщений, поэтому NOTIFY используется только как механизм пробуждения;
- при росте нагрузки можно заменить слой очереди на RabbitMQ/Kafka/Temporal, сохранив API шлюза.

## ADR-003. Lease-запись вместо долгой блокировки БД

Решение: слот удерживается через поля `lease_id`, `owner`, `expires_at`, а не через открытую транзакцию с блокировкой строки.

Причины:

- HTTP-вызов не должен держать транзакцию БД;
- долгие транзакции ухудшают работу PostgreSQL;
- откат/фиксация автоматически отпускает блокировку, но нам нужно состояние "слот занят" вне транзакции.

Обязательное правило:

```text
освобождение и продление аренды выполняются только по slot_id + lease_id
```

Это защищает от ситуации, где старый поток освобождает уже переиспользованный слот.

## ADR-004. Приоритет sync над async реализуется политикой старта

Решение:

```text
totalSlots = 5
targetFreeSyncSlots = 1
sync limit = 5
asyncAllowed = max(0, totalSlots - syncBusy - targetFreeSyncSlots)
async не стартует новые задачи при наличии живых sync waiters
```

Это скользящий sync-резерв: если sync уже занял один слот, шлюз старается оставить свободным следующий слот под еще один sync. Если активных sync-вызовов становится больше, допустимое количество новых async-вызовов уменьшается.

Причины:

- уже начатый async-вызов нельзя безопасно вытеснить;
- постоянно поддерживаемый свободный слот снижает задержку следующего sync-запроса;
- активная sync-нагрузка постепенно вытесняет старт новых async-вызовов;
- барьер ожидающих sync-запросов не дает async забрать следующий освободившийся слот.

Последствия:

- при отсутствии sync один слот может простаивать;
- при росте sync-нагрузки пропускная способность async снижается сильнее, чем при статическом лимите `async <= 4`;
- уже запущенные async-вызовы не отменяются, поэтому резерв восстанавливается только по мере их завершения;
- это осознанная цена за приоритет sync;
- политику можно усложнить позже: заимствование по времени или временное разрешение async занять последний слот после долгого отсутствия sync.

## ADR-005. Async-результат доставляется обратным вызовом, опрос остается резервным способом

Решение: для async-задач основной способ доставки результата - HTTP-обратный вызов из шлюза в сервис-клиент. Опрос через API шлюза остается резервным механизмом чтения и восстановления, а также отдельным режимом `deliveryMode=POLLING` для сервисов, которые не реализуют эндпоинт обратного вызова.

Причины:

- у нас нет Kafka/RabbitMQ и нет общей схемы БД между сервисами;
- сервис-клиент не должен регулярно опрашивать шлюз, если результат можно доставить push-моделью;
- при ошибке обратного вызова результат не теряется, потому что остается в БД шлюза;
- опрос нужен для ручного восстановления, диагностики, повторного чтения результата и клиентов без эндпоинта обратного вызова.

Последствия:

- каждый сервис-клиент с `deliveryMode=CALLBACK` должен реализовать внутренний эндпоинт обратного вызова;
- для `deliveryMode=POLLING` доставка обратного вызова не создается, а `callbackDeliveryStatus` равен `NOT_REQUIRED`;
- эндпоинт обратного вызова должен быть идемпотентным;
- шлюз должен иметь отдельные повторы и задержки между повторами для доставки обратного вызова;
- запись доставки обратного вызова создается атомарно с финальным статусом async-задачи, чтобы обратный вызов не потерялся при рестарте шлюза;
- ошибка доставки обратного вызова не меняет статус задачи внешнего сервиса.

## ADR-006. URL обратного вызова берется из списка разрешений конфигурации

Решение: шлюз не принимает произвольный `callbackUrl` в async-запросе. Эндпоинт обратного вызова выбирается по `clientService`.

Пример:

```yaml
external-gateway:
  clients:
    user-expertise:
      callback-url: http://user-expertise/internal/external-gateway/callbacks
    invest-pay:
      callback-url: http://invest-pay/internal/external-gateway/callbacks
```

Причины:

- произвольный URL в полезной нагрузке создает SSRF-риск;
- шлюз должен вызывать только доверенные внутренние сервисы;
- сетевые политики и mTLS проще настраивать на фиксированные сервисные маршруты.

Последствия:

- для нового сервиса-клиента с `deliveryMode=CALLBACK` нужна конфигурация в шлюзе;
- в запросе достаточно передать `clientService` и `deliveryMode`, но `clientService` обязательно сверяется с аутентифицированной межсервисной идентичностью.

## ADR-007. Результат внешнего сервиса нормализуется в Map<String, String>

Решение: sync-ответ, async-обратный вызов и резервный GET возвращают успешный результат как `Map<String, String>`. Для финальных неуспешных async-статусов `result` равен `null`, а причина передается в структурированном поле `error`. Строковое `lastError` может использоваться только как диагностическое краткое описание.

Причины:

- контракт простой для Java/Spring сервисов;
- результат можно безопасно хранить как JSONB;
- сервисы-клиенты не зависят от полной структуры ответа внешнего сервиса;
- в обратном вызове можно передавать результат без дополнительного запроса к шлюзу.

Последствия:

- шлюз должен иметь слой отображения ответа внешнего сервиса в `Map<String, String>`;
- если внешний сервис вернет вложенную структуру, ее нужно либо уплощить, либо явно отбросить лишние поля;
- ключи результата должны быть стабильными и документированными для конкретной операции.

## ADR-008. OpenAPI разделяется на sync, async и обратный вызов

Решение: поддерживаем отдельные OpenAPI-файлы для входящих API шлюза и исходящего контракта обратного вызова.

Причины:

- sync и async имеют разные SLA и жизненный цикл;
- async имеет статусы, резервное чтение результата, отмену и повтор;
- обратный вызов реализуют сервисы-клиенты, а вызывает его шлюз;
- потребители могут подключать только нужный контракт.

Файлы:

- `../openapi/external-gateway-sync.yaml`
- `../openapi/external-gateway-async.yaml`
- `../openapi/external-gateway-callback.yaml`

## ADR-009. API шлюза стабилен, внутренняя очередь заменяема

Решение: потребители интегрируются только через HTTP API шлюза. Они не имеют доступа к таблицам шлюза.

Причины:

- можно заменить очередь PostgreSQL на RabbitMQ/Temporal без изменения доменных сервисов;
- можно менять повторы и политику приоритета внутри шлюза;
- граница владения остается четкой.

## ADR-010. REST/OpenAPI - единственный протокол v1

Решение: v1 проектируется как REST/OpenAPI API с HTTP-обратным вызовом для доставки async-результата. Другие внутренние протоколы не входят в документацию и контракт v1.

Причины:

- проще подключить существующие Spring Boot сервисы;
- проще отлаживать через curl/Postman/логи;
- OpenAPI лучше подходит для эндпоинта обратного вызова, который должны реализовать разные сервисы;
- текущая нагрузка не требует высокой пропускной способности протокола.

Детали REST-контракта описаны в `protocol-options.md`.
