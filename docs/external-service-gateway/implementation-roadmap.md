# План реализации

Документ описывает порядок разработки `external-service-gateway`. На этом этапе код не пишется; файл нужен как чек-лист для будущей реализации.

## Шаг 0. Зафиксировать контракты

Результат:

- согласованы `../openapi/external-gateway-sync.yaml`, `../openapi/external-gateway-async.yaml` и `../openapi/external-gateway-callback.yaml`;
- определены имена сервисов-клиентов: `invest-pay`, `user-expertise`, другие будущие клиенты;
- определены эндпоинты обратного вызова для сервисов, которые хотят получать async-результат push-моделью;
- определена схема авторизации между сервисами;
- определена семантика `externalId` и `Idempotency-Key`.

Критерий готовности:

- потребители могут сгенерировать клиентов по OpenAPI;
- `user-expertise` и `invest-pay` понимают контракт эндпоинта обратного вызова;
- понятны статусы и ошибки sync/async API.

## Шаг 1. Создать Spring Boot сервис и проверить запуск (сделано)

Результат:

- текущий Spring Boot скелет проекта создан;
- добавлены эндпоинты проверки работоспособности и готовности;
- добавлена базовая конфигурация профилей.

Критерий готовности:

- сервис стартует локально и отвечает на эндпоинт дымовой проверки;
- после подключения PostgreSQL проверка готовности не проходит без доступной схемы шлюза.

## Шаг 2. Добавить PostgreSQL-схему шлюза

Результат:

- Liquibase миграции;
- таблица `ext_slots` с 5 слотами;
- таблица `ext_request_queue`;
- поле или вычисляемое значение `priority_weight` для сортировки `HIGH = 100`, `LOW = 10`;
- таблица `ext_sync_waiters`;
- таблица `ext_callback_delivery`;
- опциональная таблица `ext_call_audit`.

Критерий готовности:

- миграции создают схему с нуля;
- повторный запуск миграций идемпотентен;
- есть ограничения на статусы, приоритеты и тип слота.

## Шаг 3. Реализовать управление слотами

Результат:

- захват sync-слота: максимум 5;
- захват async-слота по динамическому sync-резерву;
- освобождение по `slot_id + lease_id`;
- продление аренды по `slot_id + lease_id`;
- восстановитель устаревших lease-записей.

Критерий готовности:

- конкурентный тест на 10+ потоков никогда не получает больше 5 слотов;
- захват async-слота соблюдает динамический лимит `asyncAllowed = max(0, 5 - syncBusy - 1)`;
- при росте `syncBusy` шлюз не стартует новые async-вызовы, пока скользящий sync-резерв не восстановится;
- старая lease-запись не может освободить новую lease-запись.

## Шаг 4. Реализовать синхронный API

Результат:

- `POST /v1/external/sync`;
- регистрация sync waiter перед ожиданием слота;
- удаление sync waiter после захвата или таймаута;
- HTTP-вызов внешнего сервиса вне транзакции БД;
- корректные ответы при таймауте, ошибке внешнего сервиса и ошибке валидации.

Критерий готовности:

- sync получает приоритет над async при освобождении слота;
- при отсутствии слота дольше SLA возвращается управляемая ошибка;
- все sync-вызовы логируются с идентификатором корреляции/запроса.

## Шаг 5. Реализовать асинхронный API

Результат:

- `POST /v1/external/async`;
- `GET /v1/external/async/{taskId}`;
- `GET /v1/external/async/by-external-id/{externalId}`;
- `DELETE /v1/external/async/{taskId}`;
- `POST /v1/external/async/{taskId}/retry`.
- поддержка `deliveryMode=CALLBACK | POLLING`;
- сохранение результата вызова внешнего сервиса как `Map<String, String>`.

Критерий готовности:

- повторная постановка с тем же `externalId` возвращает существующую задачу;
- статусы задачи отражают реальное состояние обработки;
- завершенная задача содержит результат или ошибку.
- `GET` возвращает тот же `result`, который отправляется в обратном вызове: `Map<String, String>` для `DONE` и `null` для неуспешных финальных статусов.

## Шаг 5.1. Реализовать доставку обратных вызовов

Результат:

- шлюз выбирает URL обратного вызова по `clientService` из списка разрешений конфигурации;
- шлюз не принимает произвольный `callbackUrl` из пользовательской полезной нагрузки;
- после финального статуса async-задачи шлюз отправляет обратный вызов в сервис-клиент;
- шлюз отправляет обязательный заголовок `X-Callback-Attempt` и опциональные `X-Request-Id`, `X-Gateway-Signature`;
- тело обратного вызова содержит `result` как `Map<String, String>` для `DONE` и `null` для неуспешных финальных статусов;
- доставка обратного вызова имеет собственные повторы, задержки между повторами и финальный статус `DEAD`;
- эндпоинт обратного вызова на стороне сервиса-клиента должен быть идемпотентным.

Критерий готовности:

- `user-expertise` получает обратный вызов после успешного выполнения async-задачи;
- повторная доставка одного результата не создает дублей в сервисе-клиенте;
- если обратный вызов не доставлен, результат остается доступен через `GET /v1/external/async/{taskId}`;
- ошибка доставки обратного вызова не переводит успешно выполненную задачу внешнего сервиса в `FAILED`.

## Шаг 6. Реализовать диспетчер

Результат:

- выбор задач по `priority_weight DESC, available_at ASC, id ASC`;
- `FOR UPDATE SKIP LOCKED` для захвата задачи;
- запрет старта async, если есть живые sync waiters;
- `LISTEN/NOTIFY` как механизм пробуждения;
- резервный опрос.

Критерий готовности:

- несколько инстансов шлюза не берут одну задачу дважды;
- backlog async-задач не вытесняет sync;
- при `syncBusy=1` одновременно стартует не больше 3 async-вызовов, при `syncBusy=2` - не больше 2;
- потеря NOTIFY не ломает обработку благодаря опросу.

## Шаг 7. Реализовать клиент внешнего сервиса

Результат:

- HTTP-клиент с таймаутами соединения и чтения;
- автоматический выключатель;
- политика повторов для разрешенных временных ошибок;
- запрет повторов для неидемпотентных sync-вызовов без ключа идемпотентности.

Критерий готовности:

- таймаут внешнего сервиса меньше TTL аренды слота;
- 429 от внешнего сервиса логируется как нарушение лимита или внешний конкурент;
- ошибки отображаются в понятные статусы шлюза.

## Шаг 8. Добавить восстановление

Результат:

- восстановитель для устаревших слотов;
- восстановитель для зависших задач `IN_PROGRESS`;
- перевод задач в `DEAD` после `maxAttempts`;
- перевод неретраибельных финальных ошибок в `FAILED`;
- ручной повтор из `DEAD`/`FAILED`, если финальная ошибка помечена как `retryable`.

Критерий готовности:

- рестарт шлюза во время async-вызова не теряет задачу;
- зависшая задача возвращается в обработку после TTL;
- восстановитель не освобождает чужую новую lease-запись.
- зависшая доставка обратного вызова возвращается в повтор после TTL.

## Шаг 9. Добавить наблюдаемость

Результат:

- метрики Micrometer;
- структурированные логи;
- запросы для панелей;
- оповещения.

Критерий готовности:

- видно количество занятых слотов;
- видно очередь по приоритетам;
- видны задачи в статусе `DEAD`;
- видны доставки обратных вызовов в статусах `PENDING` и `DEAD`;
- виден процент отклоненных sync-запросов.

## Шаг 10. Нагрузочные и интеграционные тесты

Минимальные сценарии:

- 20 конкурентных sync-запросов, одновременно внешний сервис получает не больше 5;
- backlog из 100 async-задач без активных sync-вызовов, одновременно внешний сервис получает не больше 4 async-вызовов;
- если активен 1 sync-вызов, async-диспетчер доводит async только до 3 одновременно выполняющихся вызовов и держит 1 свободный слот;
- если активны 2 sync-вызова, async-диспетчер доводит async только до 2 одновременно выполняющихся вызовов и держит 1 свободный слот;
- во время backlog async-задач приходит sync, следующий свободный слот получает sync;
- падение инстанса шлюза во время async-вызова;
- повторный `externalId` для async;
- успешная доставка результата обратным вызовом в `user-expertise`;
- повторный обратный вызов по той же задаче идемпотентен;
- эндпоинт обратного вызова временно недоступен, шлюз выполняет повтор, результат доступен через GET;
- `LISTEN/NOTIFY` отключен, опрос продолжает обработку.

Критерий готовности:

- тесты проходят локально и в CI;
- результаты тестов приложены к релизу v1.

## Шаг 11. Развертывание

Результат:

- шлюз разворачивается минимум в 2 инстанса;
- все потребители переключены на шлюз;
- сервисы-клиенты, использующие async-обратный вызов, открыли внутренний эндпоинт обратного вызова;
- прямые вызовы внешнего сервиса из `invest-pay` и `user-expertise` удалены или запрещены сетевой политикой.

Критерий готовности:

- внешний сервис видит вызовы только от шлюза;
- лимит 5 соблюдается в промышленной среде;
- план отката описан и проверен.
