# Детализация SDD/OpenSpec framework

Документ фиксирует изученное по папке `docs/sdd/skills`. Цель — сохранить текущую модель работы аналитика и разработки перед последующей трансформацией фреймворка и жизненного пути его этапов.

## 1. Общая модель

Фреймворк описывает файловый OpenSpec/SDD workflow для системного аналитика, архитектора/разработчика и AI-агента. OpenSpec CLI не требуется: все действия выполняются через Markdown-шаблоны, reference-файлы и прямую работу с файловой системой.

Основная идея: спецификация сервиса является живым source of truth, а любые изменения проходят через отдельный change-запрос. Для сложных изменений поверх change создаются инженерные артефакты `design.md` и `tasks.md`, затем задачи реализуются в коде, change вливается обратно в основную спецификацию и архивируется.

Ключевые слои:

- Аналитический слой: `service-spec.md`, `change.md`.
- Исследовательский слой: `.research/*.yaml`, `.research/_aggregate.yaml`, `.research-notes.md`.
- Инженерный слой: `design.md`, `tasks.md`, код проекта, сборка/тесты/линтер.
- Управляющий слой: статусы change, ручное согласование через PR, архив завершённых изменений.

## 2. Структура артефактов

Основной layout:

```text
openspec/
  specs/
    <service>/
      <service>.md
      .research/
      .research-notes.md
  changes/
    <change-name>/
      change.md
      design.md
      tasks.md
      .research/
      .research-notes.md
    archive/
      YYYY-MM-DD-<change-name>/
```

Назначение артефактов:

| Артефакт | Роль | Кто ведёт | Что содержит |
|---|---|---|---|
| `openspec/specs/<service>/<service>.md` | Основная спецификация сервиса | Аналитик / `openspec-new-spec` / `openspec-apply-change` | Самодостаточное описание поведения сервиса |
| `openspec/changes/<name>/change.md` | Предложение изменения | Аналитик / `openspec-propose` | Зачем и что меняется, без технической реализации |
| `openspec/changes/<name>/design.md` | Технический проект | Разработчик/архитектор / `openspec-design` | Как реализовать изменение в кодовой базе |
| `openspec/changes/<name>/tasks.md` | Исполняемый план | Разработчик / `openspec-design` / `openspec-implement` | Мелкие задачи-чекбоксы и чеклист верификации |
| `.research/*.yaml` | Машинные находки по ролям | `openspec-explore` или лид-скил | Структурированный read-only анализ кода |
| `.research/_aggregate.yaml` | Сводный агрегат | Лид-скил | Дедуплицированные коллекции по всем ролям |
| `.research-notes.md` | Человеко-читаемая сводка | Лид-скил | Статистика, gaps, ссылки на YAML |
| `archive/YYYY-MM-DD-<name>/` | История завершённых change | `openspec-archive-change` | Вся директория change после завершения |

## 3. Жизненный цикл изменения

Базовая цепочка:

```text
teach -> new-spec -> propose -> review/approve -> design? -> implement -> apply-change -> archive-change
                 \                                             /
                  -------- explore (read-only, optional) -------
```

Этапы:

1. `openspec-teach` загружает в контекст правила workflow, если агент впервые работает с OpenSpec или пользователь просит объяснить процесс.
2. `openspec-new-spec` создаёт первую полную спецификацию сервиса с нуля или по существующему коду, если спеки ещё нет.
3. `openspec-propose` создаёт `change.md` для изменения уже существующей спецификации.
4. Аналитик согласует change через PR и вручную переводит статус в `Согласовано`.
5. `openspec-design` опционально создаёт `design.md` и `tasks.md` для сложных изменений.
6. `openspec-implement` выполняет задачи из `tasks.md`, меняет статус change на `В реализации`, запускает обязательную верификацию.
7. `openspec-apply-change` выполняет Analyst Merge: переносит содержимое `change.md` в целевую спецификацию и переводит change в `Реализовано`.
8. `openspec-archive-change` переносит директорию change в архив и, если change был реализован, ставит `Архивировано`.

`openspec-explore` существует как standalone structured research и как prompt-library для `new-spec` / `propose`. Для размытого обсуждения он не используется.

## 4. Статусная модель

Статусы `change.md`:

| Статус | Кто ставит | Когда |
|---|---|---|
| `На согласовании` | `openspec-propose` | Change создан и отправляется на ревью |
| `Согласовано` | Аналитик вручную | PR с `change.md` вмержен, change одобрен |
| `В реализации` | `openspec-implement` | Начата разработка по `tasks.md` |
| `Реализовано` | `openspec-apply-change` | Change влит в основную спецификацию |
| `Архивировано` | `openspec-archive-change` | Change перенесён в архив |

Обратные переходы выполняются только вручную. Например, возврат из `В реализации` в `Согласовано` для доработки требований не автоматизирован.

Отдельные статусы:

- `design.md`: `Черновик`, `В разработке`, `Готово к реализации`.
- `tasks.md`: `В реализации`, `На ревью`, `Готово`.

Эти статусы являются отдельным слоем жизненного цикла реализации и ведутся разработчиком вручную. Скиллы их почти не трогают.

## 5. Роли скиллов

### `openspec-teach`

Учебник и маршрутизатор. Не создаёт артефакты, а объясняет структуру OpenSpec, статусы, mapping пользовательских запросов на нужный скилл и следующий логичный шаг.

Важные правила:

- Не использовать для реальной работы, если задача уже однозначно относится к конкретному скиллу.
- Для размытого обсуждения использовать обычный чат, а не `openspec-explore`.
- Если нет спеки сервиса, обычно сначала нужен `openspec-new-spec`.

### `openspec-explore`

Structured research кодовой базы. Работает read-only и сохраняет результаты в `.research/`.

Вход:

- `target=spec` для картирования сервиса под новую спецификацию.
- `target=change` для картирования зоны изменения.
- `service`.
- `anchor` для `target=change`.
- `thoroughness=quick|medium`.

Выход:

- `.research/<role>.yaml` по ролям.
- `.research/_aggregate.yaml`.
- `.research-notes.md`.
- Короткая сводка и список gaps для интервью.

Ограничения:

- Не редактирует код, спецификации, `change.md`, `design.md`, `tasks.md`.
- Не ведёт свободное обсуждение.
- Не запускает субагентов из субагентов.
- Лид после возврата субагентов работает с YAML-сводками, а не перечитывает код массово.

### `openspec-new-spec`

Создаёт полную спецификацию сервиса в `openspec/specs/<service>/<service>.md`.

Процесс:

1. Этап 1 интервью: ЧТО за сервис, назначение, потребители, функции, контекст, SLA, доменные сущности, FSM, инварианты.
2. Параллельное read-only исследование кода через роли `api`, `business-logic`, `data`, `integrations`, `config`, `observability`, `security`, `nfr`.
3. Этап 2 интервью: КАК сервис устроен, с вопросами по конкретным находкам из кода и `.research-notes.md`.
4. Запись спецификации по шаблону `service-spec.md`.
5. Ревью на самодостаточность, полноту, непротиворечивость, отсутствие утечек реализации.
6. Очистка `.research/` и `.research-notes.md` после подтверждения пользователя.

Ключевой guardrail: лид-скил не читает исходный код напрямую. Он оркестрирует субагентов и только точечно читает до 3 файлов для закрытия конкретного gap.

### `openspec-propose`

Создаёт `openspec/changes/<name>/change.md`.

Процесс:

1. Этап 1 интервью: ЧТО и ЗАЧЕМ меняем, мотивация, скоуп, пользователи, ожидаемый результат, ограничения.
2. Чтение целевой спецификации и structured research зоны изменения через роли `feature-scope`, `dependencies`, `cross-cutting`.
3. Этап 2 интервью: КАК именно меняем, с привязкой к текущей реализации и спецификации.
4. Проверка директории change.
5. Заполнение `change.md` по шаблону.
6. Ревью change по чеклисту.

`change.md` создаётся со статусом `На согласовании`. После merge PR аналитик вручную ставит `Согласовано`.

### `openspec-design`

Создаёт `design.md` и `tasks.md` на основе согласованного `change.md`.

Особенности:

- Используется для сложных CR; простые изменения могут пропустить этот этап.
- Structured research не запускается заново. Если рядом с `change.md` есть актуальный `.research-notes.md`, он используется как источник.
- Код изучается точечно: 3-5 файлов за раз, только затронутые узлы и ближайшие вызывающие слои.
- `design.md` обязан ссылаться на конкретные сущности кода: классы, модули, пакеты, функции или аналоги для стека.
- Нельзя придумывать технологии и библиотеки, которых нет в проекте.

`tasks.md` строится как логически сгруппированный чеклист: подготовка, core-реализация, тестирование, финализация. Каждая задача должна быть достаточно малой для одной сессии.

### `openspec-implement`

Реализует задачи из `tasks.md`.

Правила выполнения:

- `tasks.md` обязателен, `design.md` опционален.
- При статусе `Согласовано` скилл меняет только строку статуса в `change.md` на `В реализации`.
- При статусе `В реализации` продолжает с места остановки.
- При статусах `На согласовании`, `Реализовано`, `Архивировано` останавливается.
- Задачами считаются только `- [ ]` в H2-секциях вида `## <номер>. ...`.
- Чеклист верификации, статус выполнения и лог изменений в `tasks.md` не трогаются.
- После выполнения каждой задачи чекбокс сразу меняется на `- [x]`.
- Перед объявлением завершения обязательны сборка, тесты и линтер, если они применимы.

Если верификация падает, change не считается завершённым. Если задача была отмечена выполненной, но проверка показала проблему именно в ней, чекбокс возвращается в pending.

### `openspec-apply-change`

Выполняет Analyst Merge: переносит смысловые изменения из `change.md` в целевую спецификацию.

Процесс:

1. Выбрать change.
2. Проверить наличие `change.md` и статус.
3. Прочитать change и целевую спецификацию из шапки change.
4. Проверить чистоту git-состояния целевой спеки.
5. Смёржить разделы change в спецификацию по названию раздела, не по номеру.
6. Перенумеровать подразделы, обновить внутренние ссылки, пересобрать TOC.
7. Инкрементировать версию спеки.
8. Поменять только строку статуса change на `Реализовано`.
9. Выполнить семантический diff заголовков.

Откат допускается только через git, не через `.bak`.

### `openspec-archive-change`

Архивирует директорию change целиком.

Правила:

- Если имя change не указано, нужно предложить выбор активных changes.
- Если статус `Реализовано`, перед перемещением статус меняется на `Архивировано`.
- Если статус другой, архивирование возможно после подтверждения, но статус сохраняется как есть для аудита незавершённой работы.
- Директория перемещается в `openspec/changes/archive/YYYY-MM-DD-<name>/`.

## 6. Structured research

Research-контур нужен, чтобы отделить чтение кода от генерации аналитического документа и не перегружать контекст лид-агента.

Размер проекта:

| Размер | Критерий | Параллелизм |
|---|---|---|
| Малый | <50 файлов кода или 1 модуль | 2 субагента |
| Средний | 50-500 файлов или 2-10 модулей | 2-3 субагента |
| Крупный | >500 файлов или multi-module/multi-repo | 3-6 субагентов |

Роли для `target=spec`:

| Роль | Что ищет |
|---|---|
| `api` | REST/gRPC/SSE/WS/GraphQL, request/response, валидации, ошибки, хедеры, клиентский флоу |
| `business-logic` | Обработчики по триггерам, бизнес-правила, альтернативные сценарии, FSM, идемпотентность |
| `data` | Модели, таблицы, DDL/DML, миграции, Avro/Proto/JSON Schema/XSD |
| `integrations` | Kafka, gRPC, HTTP-клиенты, SOAP, БД, cron, SSE |
| `config` | Параметры конфигурации, env vars, `@ConfigurationProperties` |
| `observability` | Логи, метрики, health-check, аудит |
| `security` | Аутентификация, авторизация, TLS/mTLS, маскирование |
| `nfr` | Производительность, лимиты, пулы, таймауты, graceful shutdown, retry, circuit breaker |

Роли для `target=change`:

| Роль | Что ищет |
|---|---|
| `feature-scope` | Текущая реализация в зоне изменения, поведение обработчиков, флоу клиента, модели, правила, валидации |
| `dependencies` | Callers/потребители, breaking risk, миграция, обратная совместимость, rollback |
| `cross-cutting` | Ошибки, хедеры, безопасность, логи, метрики, конфигурация |

Инварианты research:

- Промпты ролей из `research-roles.md` копируются дословно.
- Лид только подставляет пути, anchor и thoroughness.
- Субагенты возвращают YAML с `summary`, `key_files`, доменными коллекциями и `gaps`.
- Каждый YAML валидируется; при ошибке допускается один retry.
- Дедупликация выполняется по стабильным ключам: endpoint = method+path, entity = name, metric = name, acceptance = when+then и т.д.
- Все промежуточные результаты пишутся на диск до агрегации.
- `.research-notes.md` используется как восстановление после compact и как источник вопросов для Этапа 2 интервью.

## 7. Аналитические документы и их границы

### `service-spec.md`

Спецификация — документ системного аналитика. Главный критерий: разработчик должен быть способен восстановить сервис с нуля по одной спеке без доступа к текущему коду.

Основные разделы:

1. Общие сведения: назначение, глоссарий, бизнес-инварианты.
2. Функциональные требования:
   - типовой флоу клиента;
   - обработчики по триггерам;
   - альтернативные и ошибочные сценарии;
   - бизнес-правила;
   - FSM;
   - API;
   - интеграции;
   - модели данных;
   - конфигурация;
   - логирование;
   - валидация;
   - ошибки;
   - хедеры и метаданные.
3. Рекомендации к критериям приёмки.
4. Нефункциональные требования: безопасность, производительность, надёжность, мониторинг, аудит.

Запрещено:

- Код приложения.
- Реализация методов, классов, функций.
- Unit/integration test code.
- Ссылки на классы, методы, пакеты, бины, переменные.
- Ссылки на файлы исходного кода.
- Технический псевдокод про блокировки, потоки, кеши, индексы.

Разрешено:

- Контрактные артефакты: OpenAPI, Protobuf, Avro, JSON Schema, GraphQL SDL, SQL DDL/migrations, XSD/WSDL, диаграммы поведения.
- Бизнес-правила в формате ЕСЛИ/ТОГДА/ИНАЧЕ и пошаговый бизнес-псевдокод.
- Таблицы моделей данных, API, интеграций, конфигурации.
- Критерии приёмки в формате КОГДА/ТОГДА.

### `change.md`

Change — аналитический инкремент к существующей спецификации. Он описывает ЗАЧЕМ и ЧТО меняется, но не КАК реализовать в коде.

Структура:

1. Предложение, включая цель, инициатора, компоненты, приоритет, обратную совместимость и флоу клиента.
2. Бизнес-логика.
3. Глоссарий.
4. Бизнес-инварианты.
5. Состояния и переходы FSM.
6. Изменения моделей данных.
7. Изменения интеграций.
7А. Обработчики по триггерам.
8. Обработка ошибок.
9. Хедеры и метаданные.
10. Валидация входящих значений.
11. Влияние на безопасность.
12. Миграция.
13. Логирование.
14. Мониторинг.
15. Конфигурация.
16. Рекомендации к критериям приёмки.

Для незатронутых разделов используется ровно `Нет изменений.`.

Форматы ADDED/MODIFIED/REMOVED применяются к глоссарию, инвариантам, FSM, моделям, интеграциям, обработчикам, конфигурации и флоу клиента.

### `design.md`

Design — инженерный документ. В отличие от `service-spec.md` и `change.md`, он должен ссылаться на конкретные сущности кода.

Содержит:

- контекст;
- goals / non-goals;
- технические решения и альтернативы;
- риски и mitigation;
- open questions;
- архитектуру;
- изменения в коде;
- интерфейсы и контракты;
- конфигурацию;
- детали алгоритмов;
- миграцию;
- зависимости;
- план тестирования.

### `tasks.md`

Tasks — исполняемый план. Рабочие задачи находятся только в пронумерованных H2-секциях. Нечисловые секции служат справкой и не должны автоматически отмечаться implementation-скиллом.

Содержит:

- группы задач;
- чеклист верификации по основным, альтернативным, ошибочным и нефункциональным сценариям;
- статус выполнения;
- лог изменений.

## 8. Интервью и критерии перехода

И `new-spec`, и `propose` используют двухэтапное интервью.

Для новой спеки:

1. Этап 1: понять ЧТО за сервис без привязки к коду.
2. Research кода.
3. Этап 2: уточнить КАК сервис устроен, привязывая вопросы к найденным эндпоинтам, правилам, моделям, интеграциям, конфигам.

Для change:

1. Этап 1: понять ЧТО и ЗАЧЕМ меняем без кода.
2. Research целевой спецификации и зоны изменения.
3. Этап 2: уточнить КАК именно меняется текущее поведение.

Общие правила интервью:

- Вопросы задаются порциями по 2-3.
- Нельзя создавать артефакт после одного ответа, если остаются неясности.
- Перед записью файла нужно подвести итог всех требований и получить явное подтверждение пользователя.
- Если пользователь на подтверждении добавляет новые детали, нужно снова обработать их и повторно подтвердить итог.

## 9. Качество и cross-linking

Фреймворк строит восстановимость через перекрёстные ссылки:

- Шаги клиентского флоу с изменением состояния ссылаются на переходы FSM.
- Шаги обработчиков с решающей логикой ссылаются на бизнес-правила и переходы FSM.
- Предусловия обработчиков ссылаются на инварианты или FSM.
- Ошибочные ветки ссылаются на альтернативные сценарии и коды ошибок.
- Модели данных с доменными типами ссылаются на глоссарий.
- Конфигурация, влияющая на поведение, ссылается на правила, FSM или обработчики.
- Критерии приёмки покрывают FSM и бизнес-инварианты.

Обязательные элементы:

- Каждый доменный термин должен быть в глоссарии.
- Каждый инвариант формулируется как ВСЕГДА/НИКОГДА и должен быть наблюдаемым.
- Каждое бизнес-правило имеет формулировку и пошаговый бизнес-псевдокод.
- Каждый trigger имеет обработчик с фиксированным скелетом: Триггер, Инициатор, Вход, Предусловия, Шаги, Side-effects, Выход/ответ, Ошибочные ветки, Идемпотентность.
- Для MODIFIED-обработчика нужны `было` и `стало`.
- Каждый новый/изменённый переход FSM и инвариант покрывается критерием приёмки.

## 10. Общие требования и стандарты

Из `common-requirements.md`:

- Логирование использует уровни `ERROR`, `WARN`, `INFO`, `DEBUG`, `TRACE`, `ALL`, `OFF`.
- В логах нельзя раскрывать конфиденциальную информацию K1/K2; такие данные должны маскироваться.
- Коды событий делятся по диапазонам: `1000-1999` ERROR, `2000-2999` WARN, `3000-3999` INFO, `4000-4999` DEBUG.
- HTTP-ошибки описываются в формате Problem Details, совместимом с RFC 9457/RFC 7807: `application/problem+json`, поля `type`, `title`, `status`, `errorCode`, `detail`, `instance`.
- Базовые HTTP-статусы: `400`, `403`, `404`, `500`, также встречаются `429`, `503`.
- Имена сервисов должны быть в kebab-case и отражать логическую функцию, например `<name>-service`.

Из `analytics-rules.md`:

- Документация ведётся в Git.
- Согласование идёт через Pull Request.
- В PR должны участвовать аналитик-проверяющий и разработчик.
- Коммит в master минуя PR запрещён.
- Для аналитических веток используется формат `analytics/PROJECT-XXX/<english-short-description>`.

## 11. Механика реализации и верификации

`openspec-implement` работает как executor по `tasks.md`.

Парсинг задач:

- Рабочие секции только `## 1. ...`, `## 2. ...`, и т.п.
- Первая невыполненная строка `- [ ]` становится текущей задачей.
- Нечисловые H2-секции игнорируются.

Цикл:

1. Показать текущую задачу.
2. Реализовать строго её scope.
3. Сразу отметить `- [x]`.
4. Обновить прогресс.
5. Перейти к следующей.

Пауза нужна, если:

- задача непонятна;
- обнаружен конфликт `tasks.md` и `change.md`;
- выявлена проблема в `design.md`;
- без `design.md` возник архитектурный вопрос;
- сборка/тесты/линтер падают;
- пользователь прервал работу.

Верификация определяется по признакам проекта:

- `pom.xml` -> `mvn -q verify`.
- `build.gradle` -> `./gradlew build`.
- `package-lock.json` -> `npm ci && npm run build && npm test && npm run lint`.
- `go.mod` -> `go build ./...`, `go test ./...`, `go vet ./...`.
- `pyproject.toml` / `requirements*.txt` -> `pytest`, `ruff`/`flake8`, `mypy` при наличии.
- `Cargo.toml` -> `cargo build`, `cargo test`, `cargo clippy`.

Если команды неизвестны или признаки противоречивы, нужно спросить пользователя, а не угадывать.

## 12. Механика Analyst Merge

`openspec-apply-change` переносит change в основную спецификацию.

Критические правила:

- Перед мержем целевая спецификация должна быть чистой по git.
- `.bak`-файлы запрещены.
- Разделы ищутся по названию, не по номеру.
- Разделы `Нет изменений.` пропускаются.
- Существующий незатронутый контент сохраняется.
- HTML-разметка в спеках сохраняется.
- После вставок нужно перенумеровать подразделы и обновить внутренние ссылки.
- TOC пересобирается по фактическим заголовкам.
- Версия спеки инкрементируется.
- Финальная проверка — семантический diff заголовков, а не line count.

Основной mapping change -> spec:

| `change.md` | Целевая область спеки |
|---|---|
| Бизнес-логика | Бизнес-правила и логика обработки |
| Глоссарий | Глоссарий доменных терминов |
| Бизнес-инварианты | Бизнес-инварианты |
| FSM | Состояния и переходы |
| Модели данных | Модели данных |
| Интеграции | Интеграции |
| Обработка ошибок | Обработка ошибок |
| Хедеры | Хедеры и метаданные |
| Валидация | Валидация входящих значений |
| Безопасность | НФТ / Безопасность |
| Логирование | Логирование |
| Мониторинг | НФТ / Мониторинг |
| Конфигурация | Требования к конфигурационному файлу |
| Критерии приёмки | Рекомендации к критериям приёмки |

## 13. Замеченные несостыковки и риски для трансформации

Эти пункты важно проверить перед изменением жизненного пути.

1. В разных местах встречаются разные варианты статусов: `В реализации` и `В работе`, `Архивировано` и `Заархивировано`. Нужна единая каноническая enum-модель.
2. В `openspec-design` описание требует статус `Согласовано`, но шаги явно не валидируют статус change. Для строгого процесса стоит добавить gate.
3. В `openspec-propose` роль `.research-notes.md` двойственная: `design` хочет использовать его как источник контекста, но общий `invocation-contract` говорит, что после финального артефакта research можно удалить. Нужно решить, хранить ли research до design или очищать сразу после change.
4. В `research-orchestration.md` таблица маппинга `service-spec.md` местами выглядит устаревшей по нумерации: например API указан как 2.4, тогда как актуальный шаблон содержит API в 2.5, интеграции в 2.6, модели данных в 2.7, конфигурацию в 2.8.
5. Процесс ориентируется на POSIX-команды (`mkdir -p`, `rm -rf`, `mv`, `grep`, `diff`), хотя может запускаться в Windows-среде. Для переносимости нужен shell-neutral слой команд.
6. Статус `Реализовано` трактуется как результат `apply-change`, но в некоторых формулировках звучит как состояние после реализации кода. Нужно разделить «код реализован» и «change применён к спеке» либо переименовать состояния.
7. `openspec-apply-change` допускает статусы `Согласовано` и `В реализации`, хотя семантически apply должен идти после разработки и проверки. Нужен явный policy: apply после merge кода или apply как аналитическая фиксация до/после разработки.
8. `.gitignore` для `.research/` описан в reference, но не является обязательным шагом всех скиллов. Есть риск случайного коммита временных research-артефактов.
9. `openspec-explore` standalone требует явные параметры и не поддерживает conversational-режим. Это полезно для строгости, но для будущего жизненного пути может понадобиться отдельный pre-discovery режим без записи файлов.
10. `service-spec.md` запрещает ссылки на код, а `design.md` требует ссылки на код. Это правильное разделение, но в трансформации нужно сохранить границу, иначе аналитический документ превратится в технический.

## 14. Что сохранить при трансформации

Сильные элементы текущего framework:

- Разделение source of truth, delta-change, design и tasks.
- Двухэтапное интервью: сначала смысл, потом детали после исследования.
- Read-only structured research с ролями и YAML-контрактом.
- Персистентность research на диске для восстановления после compact.
- Жёсткие guardrails против кода в аналитических документах.
- Фиксированный скелет обработчика по триггеру.
- Обязательные бизнес-правила с формулировкой и пошаговым псевдокодом.
- Cross-linking между флоу, обработчиками, правилами, FSM, инвариантами и acceptance.
- Строгий executor по `tasks.md`.
- Обязательная верификация перед объявлением реализации завершённой.
- Analyst Merge с git-точкой отката и семантическим diff заголовков.

## 15. Возможная нормализованная карта этапов

Для будущей переработки можно рассматривать такой канонический жизненный путь:

1. Discovery: понять запрос, выбрать `new-spec` или `propose`.
2. Baseline: создать или проверить актуальную спецификацию сервиса.
3. Proposal: оформить change как аналитический delta-документ.
4. Approval: PR, ревью, ручное согласование.
5. Engineering design: создать design/tasks для сложных изменений.
6. Implementation: выполнить tasks с прогрессом и верификацией.
7. Spec consolidation: применить change к source-of-truth спецификации.
8. Archive: зафиксировать историю и убрать active change.

Главный вопрос для трансформации: нужно ли сохранять линейную цепочку или ввести более явные состояния и переходы, например `Draft -> Review -> Approved -> Designed -> In Development -> Code Verified -> Spec Applied -> Archived`.
