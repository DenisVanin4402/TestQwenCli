Реализовать задачи из `tasks.md` строго по плану. Если есть `design.md`, опирайся на него; для простого CR `design.md` может отсутствовать.

**Input**: имя change. Skill автоматически использует `openspec/changes/<name>/`.

**Справочники**

- `references/runbook.md` — статусы change, парсинг tasks, цикл выполнения, output.
- `references/verification.md` — команды сборки/тестов/линтера.
- `references/edge-cases.md` — нестандартные ситуации.

---

## Steps

### 1. Выбери change

Если имя не указано — предложить выбор active changes. Объяви: `Реализую change: <name>`.

### 2. Проверь файлы

Путь:

```text
openspec/changes/<name>/
```

Обязательны:

- `change.md`;
- `tasks.md`.

Опционален:

- `design.md`.

Без `tasks.md` остановись и предложи `openspec-design`.

### 3. Прочитай change.md

Проверь:

- `> **Статус**:`;
- `> **Spec update mode**:`;
- `> **Мастер-спецификация**:`;
- `> **Manifest**:`;
- `## 0. Источники master specification`.

Статусы обрабатывай по `references/runbook.md`: `Согласовано` автоматически переводится в `В реализации`, `В реализации` продолжается, остальные статусы блокируют или требуют подтверждения.

Если `Spec update mode` отсутствует, зафиксируй риск старого шаблона и спроси пользователя, какой mode использовать для финального output.

### 4. Покажи прогресс

Посчитай выполненные и оставшиеся рабочие задачи.

Рабочими считаются только `- [ ]` / `- [x]` в H2-секциях, чей заголовок начинается с `## <номер>. `.

### 5. Прочитай design.md и master-spec context

Если `design.md` есть — изучи его.

Также прочитай из `change.md`:

- master-spec root;
- manifest;
- sources из `## 0. Источники master specification`.

Не редактируй master-spec documents в рамках implement без явного задания в рабочей задаче.

### 6. Найди первую невыполненную задачу

Ищи первую строку `- [ ]` только в рабочих H2-секциях. Нечисловые H2-секции вроде `## Чеклист верификации`, `## Статус выполнения`, `## Лог изменений` не трогай.

### 7. Выполни задачи циклом

Для каждой невыполненной задачи:

1. Покажи `Выполняю X.Y: ...`.
2. Реализуй строго scope задачи.
3. Сразу замени `- [ ]` на `- [x]`.
4. Обнови прогресс.
5. Перейди к следующей.

Пауза: непонятная задача, конфликт `tasks.md` и `change.md`, проблема в `design.md`, архитектурный вопрос без design, ошибка, прерывание пользователя.

### 8. Верификация

После выполнения всех рабочих задач и до сообщения о завершении:

- определить команды по признакам проекта;
- запустить сборку, тесты и линтер, если применимы;
- при падении перейти в pause output;
- если автоматическая верификация технически невозможна, явно зафиксировать причину.

### 9. Завершение

Если все задачи выполнены и верификация прошла:

- покажи итоговый прогресс;
- перечисли команды верификации и результат;
- покажи `Spec update mode`;
- предложи следующий шаг:
  - `branch-diff`: проверить, что документы master specification из `change.md` обновлены в ветке, затем архивировать change;
  - `manual-change`: явно обновить документы master specification или запустить `openspec-apply-change` только как manual-apply gateway;
  - mode неизвестен: попросить пользователя выбрать mode перед финальным обновлением master spec.

---

## Guardrails

- Продолжай выполнять задачи, пока они не завершены или не заблокированы.
- В `tasks.md` трогай только чекбоксы рабочих задач в пронумерованных H2-секциях.
- В `change.md` меняй только строку статуса.
- Не трогай чеклист верификации, статус выполнения и лог изменений в `tasks.md`.
- Не докладывай завершение без верификации или явной фиксации невозможности верификации.
- Не считай `openspec-apply-change` обязательным финальным шагом.
- Не редактируй master-spec documents без явной задачи или выбранного manual-change процесса.
