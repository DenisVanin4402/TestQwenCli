# Правила ведения документации

Версия спецификации - 1.2

## 1. Общие требования

Вся документация должна вестись в Git.

В качестве формата используется простой GIGA IDEA + MarkDown.
Для диаграмм последовательности GIGA IDEA + PlantUML.

## 2. Инструкция по созданию спецификации

1. В Bitbucket создается новая аналитическая ветка (Create branch -> Branch from: master -> Branch name: см. правила ниже).

2. Правило наименования веток в Git:

* при работе с фичами необходимо именовать ветку: **analytics/PROJECT-XXX/краткое описание
  задачи на английском без кавычек**
* при работе над багом: **bugfix/PROJECT-XXX/краткое описание задачи на
  английском без кавычек**

  где **PROJECT-XXX** - номер таски на аналитика в Jira/SberTrack.

3. В GIGA IDEA перейти на ветку master (Git -> выбрать master -> Checkout) и подтянуть изменения из удаленного репозитория (Git -> Update Selected).

4. Перейти на созданную ветку (Git -> выбрать ветку с PROJECT-XXX -> Checkout) и править документацию в рамках нее.

5. После готовности документации все изменения необходимо запушить в удаленную ветку (Commit -> править описание коммита (см. ниже) -> Commit and Push).

6. При commit в удаленную ветку обязательно в описании коммита указывать **PROJECT-XXX** и краткое описание доработок на русском.

7. Согласование должно проходить с использованием PullRequest. Делать commit в ветку master минуя PullRequest **запрещено**!

8. В Bitbucket создается PullRequest (Create pull request -> Source: ветка с PROJECT-XXX -> Destination: master).
   В наименовании PullRequest указать PROJECT-XXX + краткое описание на русском.
   В описании PullRequest указать всю информацию, которая необходима проверяющему для корректной валидации.
   В качестве согласующих ставятся:
   * Аналитик-проверяющий
   * Разработчик

9. После Approve от согласующих, аналитик самостоятельно делает Merge. Таска на аналитику закрывается, задача передается на разработку.

10. В случае внесения правок в текущий PullRequest достаточно запушить правки в удаленную ветки и они автоматически подтянутся в открытый PullRequest.

## 3. Расположение документов

Спеки должны лежать в следующих папках:

* **protocols/protocolName** - спека с протоколами взаимодействия;
* **services/serviceName** - спека на сервисы.

В каждой папке должны быть созданы следующие подпапки:

* **/img** - картинки по plant uml
* **/plantUML** - исходники схем для plant uml
* **/schemas** - все схемы (можно создать подпапки по
  названиям сервисов если их много; может отсутствовать)
* **/examples** - примеры сообщений (может отсутствовать или быть подпапкой schemas)
