# Interview Playbook

Двухэтапное интервью для новой спецификации сервиса.

## Этап 1 — ЧТО за сервис (без привязки к коду)

Начни с открытого вопроса:
> "Какой сервис нужно документировать? Опишите его назначение и ключевые функции."

Далее — уточняющие вопросы порциями по 2–3:

- **Назначение**: "Кто потребители этого сервиса? Какую задачу он решает для бизнеса?"
- **Функции**: "Какие ключевые операции выполняет? Что самое важное?"
- **Контекст**: "Где этот сервис в общей архитектуре? Часть платформы, standalone?"
- **Надёжность**: "Какие SLA? Что происходит при недоступности зависимостей?"
- **Доменные сущности и термины**: "Какими доменными сущностями оперирует сервис? Есть ли специфические термины (например, «резервирование», «промокод»), которые нужно зафиксировать в глоссарии?"
- **Жизненный цикл**: "Есть ли у основной сущности явные статусы и правила переходов между ними (FSM)? Какие переходы разрешены, какие — нет?"
- **Бизнес-инварианты**: "Какие правила должны выполняться ВСЕГДА (например, «сумма скидки не больше суммы заказа»)? Что НИКОГДА не должно произойти?"

**Правила этапа 1:**
- Задавай вопросы порциями по 2–3, не заваливай стеной.
- Адаптируйся к ответам — если пользователь описал подробно, не переспрашивай очевидное.
- Если ответ неполный — копай глубже.
- **НЕ переходи к этапу 2**, пока назначение и контекст сервиса не ясны.
- Когда суть понятна — подведи итог и получи подтверждение.
- Из описания выведи kebab-case имя сервиса.

## Этап 2 — КАК устроен сервис (после изучения кода)

Теперь, зная назначение и изучив код, задавай вопросы по конкретным деталям:

- **API**: "Я вижу эндпоинты X, Y, Z в коде. Все актуальны? Есть ли незадокументированные?"
- **Типовой флоу клиента (2.1.1)**: "Какой типовой порядок API-вызовов у клиента от начала сценария до завершения? Например: сначала `GET /catalog`, затем `POST /orders`, затем `POST /orders/{id}/pay` — это полный флоу? Есть ли разные сценарии для разных клиентов (покупатель / администратор / автоматика)?" Если сервис чисто событийный (только consumer-ы и cron) — зафиксируй явно: «клиентского API нет».
- **Обработчики по триггерам (2.1.2.N)**: на КАЖДЫЙ обнаруженный триггер (endpoint / Kafka-consumer / cron / gRPC / SSE / webhook) задай вопросы по скелету:
  - «Что именно делает обработчик `<триггер>` пошагово?»
  - «Какие предусловия/guard должны быть выполнены до обработки?»
  - «Где транзакционные границы? Что коммитится атомарно, а что — отдельно? Что происходит при сбое на шаге N?»
  - «Какие side-effects (запись в БД, публикация в Kafka, внешние вызовы)?»
  - «Что возвращается клиенту / публикуется наружу?»
  - «Ошибочные ветки: какие коды ошибок и ссылки на альтернативные сценарии?»
  - «Идемпотентность: по какому ключу? Что происходит при повторе — возврат исходного результата, no-op, конфликт?»
- **Бизнес-логика и псевдокод (2.3)**: "В коде я нашёл правило X. Это верно? Опиши его как ПОШАГОВЫЙ алгоритм (нумерованные шаги) на конкретном примере: какие входные значения, какие промежуточные, какой выход. Как обрабатываются края — null, пустой ввод, округления, гонки?" Добивайся пошагового псевдокода для КАЖДОГО правила, даже линейного.
- **Бизнес-формулы**: "В коде расчёт суммы/скидки/тарифа выглядит как F(...). Это бизнес-формула в актуальной редакции? Округление, краевые значения? Покажи расчёт пошагово на конкретных числах."
- **FSM**: "В коде вижу переходы статусов A → B → C. Подтверди полный список разрешённых переходов и явно перечислим запрещённые. Что является триггером каждого перехода?"
- **Инварианты**: "Какие правила должны выполняться ВСЕГДА для основной сущности? Например, найденное в коде ограничение X — это инвариант или частный случай?"
- **Глоссарий**: "Термины X и Y используются в коде в разных местах — это одно и то же или разные понятия? Зафиксируем определение."
- **Интеграции**: "Сервис вызывает A и B. Это полный список? Как обрабатываются ошибки вызовов?"
- **Данные**: "Вижу сущности X, Y. Какие связи между ними? Есть ли кэширование?"
- **Авторизация**: "Какие роли имеют доступ? Есть ли ограничения по данным?"
- **Конфигурация**: "Вижу параметры X, Y в конфиге. Все документировать? Есть ли feature-флаги?"

**Правила этапа 2:**
- Привязывай вопросы к конкретике из кода — "Я вижу в коде X. Это актуально?"
- Если нашёл несоответствия или неочевидную логику — уточни.
- **НЕ переходи к созданию файла**, пока остаются неточности, неясности или недосказанности.
- Когда все детали разобраны — подведи итог ВСЕХ собранных данных и явно спроси: "Всё ли учтено? Есть ли что-то, что я упустил?"
- Только после подтверждения — переходи к созданию спецификации.
- Если при подтверждении пользователь добавляет новые детали — обработай их и снова подведи итог.
