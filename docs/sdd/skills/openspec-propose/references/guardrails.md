# Guardrails — qualitative checklist для change.md

Подробный чеклист качества и запретов. Применяй перед фиксацией change.md и при ревью.

---

## Роль документа — КРИТИЧНО

`change.md` — документ **системного аналитика**, а не разработчика.
Описывай **ЧТО** должно измениться и **ЗАЧЕМ**, но **НИКОГДА** не пиши **КАК** реализовать в коде.

### ЗАПРЕЩЕНО включать в change.md

- Код на любом языке приложения (Java, Kotlin, Python, Go, JS/TS и т.д.)
- Реализацию методов, классов, функций
- Unit-тесты, integration-тесты, тестовый код любого вида
- Конкретные алгоритмические решения — это задача разработчика
- План внедрения с техническими шагами реализации
- Диаграммы классов, code-review замечания
- Конфигурационные файлы целиком (`application.yaml`, `.properties`, `.env`)
- **Ссылки на объекты исходного кода**: имена классов (`OrderService`), методов (`createOrder()`), пакетов (`com.shop.orders`), бинов, модулей
- **Ссылки на файлы исходного кода**: `src/main/java/...`, `pkg/handlers/...`, `app/services/...` и подобные пути к реализации

### РАЗРЕШЕНО

- Бизнес-правила (раздел 2) — каждое правило ОБЯЗАНО содержать (1) формулировку **ЕСЛИ/ТОГДА/ИНАЧЕ** и/или математику и (2) **пошаговый бизнес-псевдокод** с явной обработкой краёв (null, округления, гонки) — на языке, независимом от реализации (без Java/Python/Go-синтаксиса, без аннотаций фреймворков, без обращений к коллекциям, потокам, кешам, индексам БД). Технический псевдокод — запрещён, это территория архитектора.
- **Обработчики по триггерам (раздел 7А)** — на каждый новый/изменяемый триггер (HTTP endpoint / Kafka-consumer / cron / gRPC / SSE / webhook) фиксированный скелет: Триггер, Инициатор, Вход, Предусловия (guard), Шаги обработки, Side-effects, Выход/ответ, Ошибочные ветки, Идемпотентность. Для MODIFIED — обязательно «Шаги (было)» и «Шаги (стало)» либо явный diff по полям скелета. Допустимо указывать транзакционную границу на бизнес-уровне («атомарно в транзакции», «outbox», «saga») без упоминания конкретных фреймворков.
- **Флоу клиента (подраздел 1.5)** — изменения в типовой последовательности API-вызовов клиента: таблицы ADDED/MODIFIED/REMOVED. Если флоу не меняется — «Нет изменений».
- Модели данных как **таблицы полей** (имя, тип, обязательность, пример, описание)
- API как описание эндпоинтов (метод, путь, request/response **схема**), а не реализация
- Рекомендации к критериям приёмки в формате **КОГДА/ТОГДА** (без тестового кода)
- Конфигурация как **таблица параметров**, а не как yaml/properties-файл
- **SQL** — ТОЛЬКО как рекомендации по DDL/DML (`CREATE TABLE`, `ALTER TABLE`) для структуры данных
- **JSON-schema**, **Protobuf**, **Avro** — для описания моделей данных
- Описательные диаграммы потоков/состояний (PlantUML/Mermaid) — если они про домен, не про реализацию

### Допустимые ссылки на артефакты-контракты

Change **может и должен** ссылаться на контрактные артефакты (они описывают поведение, а не реализацию):

- **OpenAPI/Swagger**, **Protobuf**, **Avro**, **JSON Schema**, **GraphQL SDL**
- **SQL DDL / migrations** (`migration/V3__add_promo_codes.sql`, `db/changelog.yaml`)
- **XSD / WSDL**
- **Диаграммы поведения**: PlantUML/Mermaid (sequence, activity, state) — про домен, не про классы

**Ключевое отличие**: контракт описывает **поведение интерфейса** — он самодостаточен. Класс/метод/файл исходного кода описывает **реализацию** — без кода бесполезен.

### Если пользователь просит добавить код

Объясни: change.md описывает **требования**, а реализация — задача разработчика на основе этих требований. Реализация и код — в следующих скиллах (`openspec-design`, `openspec-implement`).

## Процесс — обязательные правила

- **НИКОГДА** не создавай change.md, пока остаются неясности — проведи полноценное интервью (см. `interview-playbook.md`) до полной ясности.
- **ВСЕГДА** изучи целевую спецификацию **ПЕРЕД** созданием change.md.
- Перед созданием файла **подведи итог ВСЕХ требований** и получи **явное подтверждение** пользователя.
- **НЕ копируй** примеры из шаблона/references в итоговый файл.
- Каждый заполненный раздел должен содержать **реальные данные**, а не плейсхолдеры и не TODO.
- Формулировки **ДОЛЖЕН/ОБЯЗАН** для обязательных требований; **СЛЕДУЕТ** для рекомендованных; **МОЖЕТ** для опциональных.
- Если change с таким именем уже существует — **спроси**, продолжить его или создать новый.
- После создания проверь: файл существует, все 16 основных разделов плюс 1.5 (Флоу клиента) и 7А (Обработчики) на месте, незатронутые помечены `Нет изменений.`.

## Ревью change.md (чеклист)

1. **Полнота** — все ли аспекты задачи покрыты? Нет ли пропущенных разделов, которые должны быть заполнены?
2. **Точность** — соответствуют ли описанные изменения наблюдаемому поведению текущего сервиса и согласуются ли с целевой спекой?
3. **Непротиворечивость** — нет ли конфликтов между разделами внутри change.md?
4. **Конкретность** — нет ли размытых формулировок, плейсхолдеров, TODO?
5. **Реализуемость** — возможно ли реализовать описанное? Нет ли пропущенных зависимостей?
6. **Границы роли** — нет ли кода приложения, unit-тестов, реализационных деталей? Нет ли ссылок на классы/методы/пакеты/файлы исходного кода? Ссылки допустимы только на контрактные артефакты (OpenAPI/proto/avro/SQL DDL/диаграммы поведения). Бизнес-псевдокод в разделе 2 — разрешён и требуется; технический псевдокод (блокировки, потоки, кеши, индексы БД) — запрещён.
7. **Форматы** — ЕСЛИ/ТОГДА/ИНАЧЕ + пошаговый псевдокод для бизнес-правил, КОГДА/ТОГДА для критериев приёмки.
8. **Флоу клиента (1.5)** — если change затрагивает порядок API-вызовов клиента, подраздел заполнен таблицами ADDED/MODIFIED/REMOVED; иначе — явное «Нет изменений».
9. **Обработчики (7А)** — если change добавляет/меняет/удаляет триггер (endpoint/consumer/cron/gRPC/SSE/webhook), раздел 7А содержит соответствующий подраздел со ВСЕМИ полями скелета (Триггер, Инициатор, Вход, Предусловия, Шаги, Side-effects, Выход, Ошибочные ветки, Идемпотентность). Для MODIFIED — есть «Шаги (было)» и «Шаги (стало)».

Верни результат: список найденных проблем или «Ревью пройдено, замечаний нет».
