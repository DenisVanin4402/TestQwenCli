# Section Guide — заполнение change.md

Детальный guide по каждому разделу шаблона `../templates/change.md` (16 основных разделов + подраздел 1.5 «Флоу клиента» + дополнительный раздел 7А «Обработчики по триггерам»). Используй при заполнении: что ожидается, частые ошибки, форматы. Реальные сниппеты и форматы — в `change-examples.md` и `example-change.md` (рядом с этим файлом).

---

## Общие правила для всех разделов

- Если раздел **не затрагивается** изменением — пиши ровно одну строку: `Нет изменений.`
- Если используешь шаблон — **удали ВСЕ HTML-комментарии** (`<!-- -->`) из итогового файла.
- Примеры из соседних reference-файлов берутся **только как ориентир формата** — **никогда не копируй содержимое** примеров в итоговый файл.
- Заполненный раздел содержит **реальные данные текущего change**, без плейсхолдеров и TODO.

## Цветовые маркеры (Markdown/HTML)

- `<font color="red">**Alt**</font>` — альтернативный/ошибочный поток
- `<font color="blue">**Opt**</font>` — опциональный элемент
- `<font color="blue">**Loop**</font>` — циклический элемент
- `<font color="green">**Примечание:**</font>` — важные примечания

## Уровни обязательности требований

- **ДОЛЖЕН / ОБЯЗАН** — обязательно, система не работает без этого
- **СЛЕДУЕТ** — рекомендуется, но допускается отступление с обоснованием
- **МОЖЕТ** — опционально

---

## Шапка (метаданные)

Заполни: имя change, статус (`На согласовании` при создании), дата, автор, `Мастер-спецификация`, `Manifest`, `Spec update mode`. Остальные статусы (`Согласовано`, `В реализации`, `Реализовано`, `Архивировано`) проставляются вручную в дальнейшем workflow.

## 1. Предложение

Ожидается: цель изменения (ЗАЧЕМ, бизнес-мотивация), инициатор, затронутые компоненты, приоритет, обратная совместимость. Дополнительно — **Флоу клиента (ADDED/MODIFIED/REMOVED)** в формате трёх таблиц: изменения в типовой последовательности API-вызовов клиента из master specification. Если флоу не меняется — одна строка «Нет изменений».
Частые ошибки: технические детали в цели вместо бизнес-мотивации; неопределённое «нужно улучшить X»; пропущенный инициатор; не отмечены компоненты влияния; флоу клиента не заполнен, когда change добавляет/удаляет endpoint или меняет порядок вызовов.
Формат цели: 2–5 абзацев на человеческом языке, без кода.

## 2. Бизнес-логика

Ожидается: пронумерованные правила. Каждое правило — ДВЕ части: (1) ФОРМУЛИРОВКА в формате ЕСЛИ/ТОГДА/ИНАЧЕ и/или математическая запись; (2) ПОШАГОВЫЙ БИЗНЕС-ПСЕВДОКОД — нумерованные шаги с явной обработкой краёв (null, пустой ввод, округления, гонки). Псевдокод ОБЯЗАТЕЛЕН для КАЖДОГО правила (даже линейного). Язык — независимый от языка реализации.
Частые ошибки: описание кода вместо правил; отсутствие пошагового псевдокода; пропущенные крайние случаи (null, округление, гонки); правила перехода статусов внутри 2 (они должны быть в разделе 5 FSM).

## 3. Глоссарий

Ожидается: ADDED / MODIFIED / REMOVED для доменных терминов. Термины из таблицы должны соответствовать глоссарию master specification.
Частые ошибки: добавление термина без определения; разные определения одного термина в разных разделах change.md.

## 4. Бизнес-инварианты

Ожидается: ADDED / MODIFIED / REMOVED для нерушимых правил. Формулировка ВСЕГДА / НИКОГДА. Каждый инвариант наблюдаем (можно проверить тестом или запросом). В колонке «Покрывающий критерий приёмки» — ссылка на № в разделе 16.
Частые ошибки: размытая формулировка («нужно обеспечивать корректность»); инвариант без покрытия в критериях приёмки.

## 5. Состояния и переходы (FSM)

Ожидается: ADDED / MODIFIED / REMOVED для состояний и переходов. Таблица переходов: #, сущность, FROM, TO, триггер, guard (со ссылкой на правило в 2), side-effect. Отдельно — изменения списка запрещённых переходов.
Если сервис не управляет жизненным циклом сущности — пишется «Нет изменений».
Частые ошибки: добавление перехода без триггера и guard; решающая бизнес-логика внутри FSM (она должна быть в 2 + ссылка из guard).

## 6. Модели данных

Ожидается: таблицы полей (имя, тип, обязательность, пример значения, описание). JSON-schema/Protobuf/Avro допустимы.
SQL — ТОЛЬКО как DDL/DML-рекомендации (`CREATE TABLE`, `ALTER TABLE`) для описания структуры.
Все доменные термины в моделях должны соответствовать глоссарию (раздел 3).
Частые ошибки: вставка Java/Python-классов с аннотациями.

## 7. Интеграции

Ожидается: эндпоинты (метод, путь), request/response схемы, коды ответов, заголовки, примеры полезной нагрузки. Для внешних интеграций — контракт и направление вызова, таймаут, retry.
Частые ошибки: реализация контроллеров вместо контрактов; смешивание контракта и поведения обработчика (поведение идёт в 7А).

## 7А. Обработчики (по триггерам)

Ожидается: изменения в обработчиках по триггерам из master specification. Три секции — ADDED / MODIFIED / REMOVED. На каждый триггер (HTTP endpoint / Kafka-consumer / cron / gRPC / SSE / webhook) — отдельный подраздел с ФИКСИРОВАННЫМ скелетом: **Триггер, Инициатор, Вход, Предусловия (guard), Шаги обработки, Side-effects, Выход/ответ, Ошибочные ветки, Идемпотентность**. Для MODIFIED — обязательно «Шаги (было)» и «Шаги (стало)» либо явный diff по конкретным полям скелета. Если изменения не затрагивают обработчики — одна строка «Нет изменений».
Частые ошибки: пропущен какой-то триггер, затрагиваемый change; только описание контракта без пошагового поведения (контракт идёт в 7, поведение — в 7А); отсутствие поля «Идемпотентность»; для MODIFIED — отсутствие «было/стало».

## 8. Обработка ошибок

Ожидается: перечень ошибок (код, HTTP-статус, errorCode, ситуация), формат по RFC 7807 из `common-requirements.md`.
Частые ошибки: код `try/catch` вместо описания.

## 9. Хедеры и метаданные

Ожидается: таблица операций ADDED/MODIFIED/REMOVED по транспортам (HTTP/Kafka/gRPC/SSE/MCP).
Частые ошибки: смешение request/response направления; отсутствие назначения хедера.

## 10. Валидация входящих значений

Ожидается: таблица «Поле / Тип валидации / Правило / Сообщение / Код ошибки».
Частые ошибки: валидация без привязки к коду ошибки из раздела 8.

## 11. Безопасность и доступ

Ожидается: изменения ролей, permissions, аудита, PII/ПДн, шифрования.
Частые ошибки: молча «не меняется» без анализа; забытое разграничение доступа к новым полям.

## 12. Миграция

Ожидается: что мигрируется, стратегия (онлайн/офлайн), обратная совместимость API, rollback-план.
Частые ошибки: отсутствие rollback; скрипты миграции в теле change.md.

## 13. Логирование

Ожидается: новые/изменённые события с уровнем (по `common-requirements.md` раздел 1), кодом, форматом сообщения.
Частые ошибки: «добавить логи» без указания уровня и полей.

## 14. Мониторинг

Ожидается: новые метрики (имя, тип, теги), новые вычисляемые метрики (PromQL).
Частые ошибки: метрика без тегов; формула без описания семантики.

## 15. Изменения конфигурации

Ожидается: таблицы ADDED / MODIFIED / REMOVED параметров (имя, тип, значение по умолчанию, описание, обязательность).
Частые ошибки: yaml/properties-блоки целиком — запрещено; только таблица.

## 16. Рекомендации к критериям приёмки

Ожидается: сценарии в формате КОГДА/ТОГДА. Каждый новый/изменённый переход FSM (раздел 5) и каждый новый/изменённый инвариант (раздел 4) ДОЛЖНЫ иметь хотя бы один критерий приёмки.
Частые ошибки: unit-тесты/код тестов внутри — запрещено; должны быть поведенческие сценарии; пропуск покрытия для FSM/инвариантов.
Формат: см. «16. Рекомендации к критериям приёмки» в `change-examples.md`.

---

## Большие файлы

Change.md может получиться объёмным (200+ строк). Если файл большой — записывай его **по частям** (раздел за разделом), чтобы не потерять контент из-за ограничений контекстного окна. Предупреди пользователя: «Change объёмный, записываю по частям».
