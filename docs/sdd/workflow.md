# Workflow SDD

Документ описывает практический процесс Specification-Driven Development для данного проекта.

## Обзор процесса

Полный цикл SDD-задачи:

```
Intake -> Specify -> Clarify -> Plan -> Decide(ADR) -> Task -> Test-first -> Implement -> Evaluate -> Learn
```

## Фазы

### 1. Intake — Сбор требований

**Цель:** Зафиксировать бизнес-цель, пользователя, scope и non-scope.

**Вход:** User intent, issue, или запрос в чат.

**Артефакт:** `docs/specs/<id>-<slug>/spec.md` (секции problem, users, goals, non-goals)

**Gate:** Product/owner approval — пользователь подтвердил, что задача охватывает нужную проблему.

### 2. Specify — Спецификация

**Цель:** Описать поведение без описания реализации.

**Вход:** Intake-документ или user request.

**Артефакты:**
- `spec.md` — спецификация с user stories, requirements (REQ-*), acceptance criteria (AC-*)
- `requirements.md` — детальные требования в формате EARS/Gherkin, если нужно

**Правила:**
- Каждый requirement получает уникальный ID (REQ-1, REQ-2, ...)
- Каждый acceptance criterion получает ID (AC-1, AC-2, ...)
- NFR (безопасность, приватность, производительность) включаются в спецификацию
- Неоднозначности помечаются `[NEEDS CLARIFICATION]`

**Gate:** Нет `[NEEDS CLARIFICATION]` для критических путей.

### 3. Clarify — Уточнение

**Цель:** Закрыть все неоднозначности.

**Вход:** `spec.md` с `[NEEDS CLARIFICATION]` вопросами.

**Артефакт:** `clarifications.md` или секция в spec.md с ответами.

**Правила:**
- Все `[NEEDS CLARIFICATION]` должны получить ответ от пользователя
- Не уточнённые требования блокируют переход к реализации

**Gate:** Все `[NEEDS CLARIFICATION]` удалены или явно перенесены как follow-up.

### 4. Plan — Планирование

**Цель:** Выбрать технический подход.

**Вход:** Утверждённая спецификация.

**Артефакты:**
- `plan.md` — технический подход, задействованные компоненты
- `research.md` — альтернативы и ограничения
- `data-model.md` — сущности, инварианты, миграции (если применимо)

**Gate:** Technical review пользователем.

### 5. Decide — ADR (при необходимости)

**Цель:** Зафиксировать значимые архитектурные решения.

**Вход:** Технические trade-offs из plan.md.

**Артефакт:** `docs/adr/<id>-<title>.md`

**Правила:**
- ADR требуется для: выбора технологии/библиотеки, изменения public API, boundary решений, структурных изменений

**Gate:** ADR accepted.

### 6. Task — Разбиение задач

**Цель:** Разбить работу на проверяемые шаги.

**Вход:** plan.md, ADR (если есть), spec.md.

**Артефакт:** `tasks.md` — граф задач с зависимостями и владельцами.

**Правила:**
- Каждая задача ссылается на REQ-ID
- Каждая задача имеет чёткий вход и выход
- Зависимости задач явные

**Gate:** Dependencies и ownership ясны.

### 7. Test-first — Сначала тесты

**Цель:** Сформировать тесты/контракты до реализации.

**Вход:** spec.md, acceptance criteria, contracts.

**Артефакты:**
- `test-plan.md` — план тестирования
- Тестовые файлы с ожидаемым падением (expected failing)
- Обновления контрактов в `contracts/`

**Правила:**
- Критические acceptance criteria имеют покрытие тестами
- Тесты пишутся ДО реализации

**Gate:** Тесты написаны и явно падают или есть explicit exception.

### 8. Implement — Реализация

**Цель:** Изменить код малыми партиями.

**Вход:** tasks.md, test-first артефакты.

**Артефакты:**
- Изменённые файлы кода
- Обновлённый `task-state.md`
- Записи в `work-log.md`

**Правила:**
- Минимальные изменения, охватывающие текущую задачу
- Запуск verification команд после каждого изменения

**Gate:** Relevant checks pass.

### 9. Evaluate — Верификация

**Цель:** Независимо проверить соответствие spec.

**Вход:** diff, spec.md, тесты, contracts.

**Артефакты:**
- Матрица трассировки (traceability matrix)
- Обзор findings в `work-log.md`

**Gate:** Все findings resolved или accepted.

### 10. Learn — Обновление памяти

**Цель:** Обновить правила и память.

**Вход:** Результаты выполненной задачи.

**Артефакты:**
- Обновлённый `AGENTS.md` (если есть стабильные правила)
- Обновлённый `QWEN.md` (Qwen-specific)
- Новые skills/commands (если применимо)
- Обновлённые ADR

**Gate:** Maintainer review.

## Команды Qwen CLI для SDD

Проект поддерживает следующий набор команд для SDD-рабочего процесса:

| Команда | Описание |
|---------|----------|
| `/sdd:specify` | Создать или обновить спецификацию |
| `/sdd:clarify` | Сгенерировать вопросы для уточнения требований |
| `/sdd:plan` | Создать технический план из спецификации |
| `/sdd:tasks` | Разбить план на проверяемые задачи |
| `/sdd:implement` | Реализовать задачу с верификацией |
| `/sdd:review` | Ревью реализации против спецификации |

Эти команды определены в `.qwen/commands/sdd/`.
