# Технический план: SDD from Diff

## Обзор

Добавить сценарий «Spec из diff MR аналитика» в существующий SDD-фреймворк. Все артефакты — Markdown + shell-скрипты. Нет изменений к Java-коду приложения.

## Подход

Последовательное создание файлов по фазам: сначала шаблоны, затем скрипты diff-extractor, потом Qwen команда/навык, затем обновление workflow/instructions.

## Затронутые компоненты

| Компонент | Изменение |
|-----------|-----------|
| `docs/specs/_template/delta-spec.md` | новый шаблон |
| `scripts/sdd-diff-extract.sh` | новый bash-скрипт |
| `scripts/sdd-diff-extract.bat` | новый bat-скрипт |
| `.qwen/commands/sdd/from-diff.md` | новая команда |
| `.qwen/skills/sdd-diff-analyzer/` | новый навык (SKILL.md) |
| `docs/sdd/workflow.md` | обновление — новый entrypoint |
| `AGENTS.md` | обновление — delta-spec правила |
| `docs/sdd/instructions.md` | обновление — новый сценарий |
| `src/` | no change |
| `pom.xml` | no change |

## Альтернативы

### Альтернатива 1: Интеграция в существующий `/sdd:specify`
- **Плюсы:** Одна команда, меньше файлов
- **Минусы:** Смешивает два разных workflow (from-prompt vs from-diff), сложнее контекст

### Альтернатива 2: Отдельный репозиторий/плагин
- **Плюсы:** Изоляция
- **Минус:** Дублирует контекст, сложнее использовать

**Выбрано:** Новая команда `/sdd:from-diff` + новые артефакты в том же репозитории, т.к. это part of the same SDD system.

## Зависимости

- git (для diff-extractor)
- Нет внешних зависимостей

## Риски

- Сложность парсинга YAML/Diff на Windows — bash/bat скрипты. Решение: минимальный парсинг, основная работа — через чтение файлов.
- Большие MR (>50 файлов) — diff-summary может стать огромным. Решение: limit 50 files, truncate with warning.
- Неполный контекст — если аналитик изменил файлы но не описал изменения явно. Решение: impact-map показывает что связано, но требует дополнения от пользователя.
