# Clarifications: 0003-task-crud

## Открытые вопросы

### CL-1: Формат UUID в URL
**Связано с:** REQ-3, REQ-4, REQ-5
**Контекст:** Эндпоинты принимают `{id}` в URL. Неясно, требуется ли строгий UUID-формат.

**Варианты:**
- **A.** Строгий UUID: если путь не соответствует формату UUID (8-4-4-4-12 hex), вернуть 400 Bad Request. Если UUID корректен но задача не найдена — 404.
- **B.** Любой строковый ID: формат не валидируется, только lookup. Если не найдено — 404.

**Рекомендация:** Вариант A — раннее обнаружение ошибок клиента, более строгий контракт.

**Blocking?** Да — влияет на тесты (400 vs 404) и валидацию.

### CL-2: PATCH для частичного обновления
**Связано с:** REQ-4 (scope)
**Контекст:** В текущей спеке только PUT (полное обновление). Нужен ли PATCH?

**Варианты:**
- **A.** Не включать PATCH в v1. Только PUT — клиент отправляет все поля.
- **B.** Добавить PATCH /api/v1/tasks/{id} — обновление только указанных полей (например, смена статуса без пересылки title).

**Рекомендация:** Вариант A — v1 минимальный, PATCH можно добавить позже.

**Blocking?** Да — влияет на scope задач и endpoints.

### CL-3: Формат ответа на DELETE
**Связано с:** AC-10
**Контекст:** Удаление задачи — какой код и тело ответа?

**Варианты:**
- **A.** 204 No Content — без тела ответа (RESTful default для DELETE).
- **B.** 200 OK с JSON удалённой задачи — клиент получает подтверждение что именно было удалено.

**Рекомендация:** Вариант A — соответствует REST конвенциям, AC-10 уже написан так.

**Blocking?** Да — фиксирует контракт ответа.

---

### CL-4: Формат времени createdAt / updatedAt
**Связано с:** REQ-1, REQ-4 (модель данных)
**Контекст:** В каком формате сериализовать время в JSON?

**Варианты:**
- **A.** ISO-8601 UTC: `"2024-01-15T10:30:00Z"` — человекочитаемый, стандарт JSON-API.
- **B.** Unix timestamp (ms): `1705312200000` — компактный, но менее читаемый.

**Рекомендация:** Вариант A — ISO-8601 UTC, стандарт де-факто для REST API, Jackson по умолчанию его использует.

**Blocking?** Да — влияет на модель и тесты.

---

### CL-5: Формат JSON-тела ошибок
**Связано с:** REQ-6, REQ-7 (AC-2, AC-3, AC-4, AC-7, AC-9, AC-11)
**Контекст:** В каком формате возвращать детали ошибок 400/404?

**Варианты:**
- **A.** Структурированный JSON:
  ```json
  {
    "error": "Validation failed",
    "message": "title must not be blank",
    "status": 400,
    "timestamp": "2024-01-15T10:30:00Z"
  }
  ```
- **B.** Минимальный JSON: `{"error": "title must not be blank"}`
- **C.** Plain text response body (без JSON Content-Type для ошибок)

**Рекомендация:** Вариант A — структурированный, позволяет клиенту программatically обрабатывать ошибки. Совместим со Spring Boot `@ControllerAdvice`.

**Blocking?** Да — влияет на тесты и контракт обработки ошибок.
