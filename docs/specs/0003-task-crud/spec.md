
# Спецификация: REST API для управления задачами (Task Management CRUD)

## Проблема

Приложение не предоставляет программных интерфейсов для создания, чтения, обновления и удаления сущностей «задача» (task). Без REST API невозможно интегрировать данное приложение с внешними системами или создать клиентское приложение, управляющее задачами.

## Цель

Реализовать полный CRUD REST API для управления задачами с валидацией входных данных, поиском по идентификатору и списком всех задач, а также возможностью удаления.

## Пользователи

- Клиентские приложения, взаимодействующие с REST API
- Внешние системы, интегрированные через HTTP
- Разработчики и тестировщики, использующие API для проверки функциональности

## Scope

**Включает:**
- REST API endpoints: POST /api/v1/tasks, GET /api/v1/tasks, GET /api/v1/tasks/{id}, PUT /api/v1/tasks/{id}, DELETE /api/v1/tasks/{id}
- Модель задачи с полями: id (UUID), title, description (optional), status, createdAt, updatedAt
- Валидация входных данных (title обязателен, макс. длина 255 символов; status в допустимом наборе)
- In-memory хранилище (List/Map) — база данных НЕ подключается
- HTTP-коды: 201 (created), 200 (ok / deleted), 400 (bad request / invalid UUID), 404 (not found)
- Валидация формат UUID в URL: не-UUID → 400
- Формат ошибок: `{"error", "message", "status", "timestamp"}`
- Формат времени: ISO-8601 UTC (`2024-01-15T10:30:00Z`)

**Не включает (non-goals):**
- Подключение к базе данных (H2, PostgreSQL и т.д.)
- Пагинация, фильтрация, сортировка списка задач
- Авторизация и аутентификация
- Категории, теги, подзадачи, приоритеты
- Soft delete (удаление жёсткое)

## User Stories

- US-1: Как пользователь, я хочу создать задачу с заголовком, чтобы зафиксировать дело, которое нужно выполнить.
- US-2: Как пользователь, я хочу получить список всех задач, чтобы увидеть общую картину.
- US-3: Как пользователь, я хочу обновить статус задачи, чтобы отразить прогресс.
- US-4: Как пользователь, я хочу удалить задачу, чтобы убрать неактуальные дела.
- US-5: Как пользователь, я хочу получить конкретную задачу по ID, чтобы просмотреть её детали.

## Требования

- **REQ-1:** POST /api/v1/tasks — создание новой задачи
- **REQ-2:** GET /api/v1/tasks — получение списка всех задач
- **REQ-3:** GET /api/v1/tasks/{id} — получение задачи по идентификатору
- **REQ-4:** PUT /api/v1/tasks/{id} — полное обновление задачи
- **REQ-5:** DELETE /api/v1/tasks/{id} — удаление задачи
- **REQ-6:** Валидация входных данных для создания/обновления
- **REQ-7:** Возврат корректных HTTP-кодов и заголовков
- **REQ-8:** In-memory хранилище (без подключения к БД)

## Acceptance Criteria

- **AC-1 (REQ-1):** POST /api/v1/tasks с корректным телом `{"title": "string", "description": "string"}` возвращает 201 и JSON созданной задачи с автоматически сгенерированным UUID `id`, статусом `TODO` и полями `createdAt`/`updatedAt`.
- **AC-2 (REQ-1):** POST /api/v1/tasks без `title` или с пустым `title` возвращает 400.
- **AC-3 (REQ-1):** POST /api/v1/tasks с `title` длиннее 255 символов возвращает 400.
- **AC-4 (REQ-1):** POST /api/v1/tasks с неизвестным `status` возвращает 400.
- **AC-5 (REQ-2):** GET /api/v1/tasks возвращает 200 и массив JSON всех задач (пустой массив, если задач нет).
- **AC-6 (REQ-3):** GET /api/v1/tasks/{id} возвращает 200 и JSON задачи, когда задача существует.
- **AC-7 (REQ-3):** GET /api/v1/tasks/{id} возвращает 404, когда задача не найдена.
- **AC-8 (REQ-4):** PUT /api/v1/tasks/{id} с корректным телом возвращает 200 и JSON обновлённой задачи с обновлённым `updatedAt`. `createdAt` остаётся неизменным.
- **AC-9 (REQ-4):** PUT /api/v1/tasks/{id} возвращает 404, когда задача не найдена.
- **AC-10 (REQ-5):** DELETE /api/v1/tasks/{id} возвращает 200 и JSON удалённой задачи, когда задача найдена и удалена.
- **AC-11 (REQ-5):** DELETE /api/v1/tasks/{id} возвращает 404, когда задача не найдена.
- **AC-12 (REQ-6):** {id} в URL должен быть в строгом UUID-формате (8-4-4-4-12 hex). Не-UUID значение возвращает 400.
- **AC-13 (REQ-7):** Все JSON-ответы (включая ошибки) имеют `Content-Type: application/json` (кроме 3xx redirect).
- **AC-14 (REQ-7):** Ответы ошибок (400, 404) содержат структурированное JSON-тело:
  ```json
  {
    "error": "краткий код ошибки",
    "message": "человекочитаемое описание",
    "status": 400,
    "timestamp": "2024-01-15T10:30:00Z"
  }
  ```

## Требования в формате EARS

- **Ubiquitous:** Система должна возвращать JSON для всех GET и POST/PUT ответов.
- **Event-driven:** Когда получен POST-запрос на `/api/v1/tasks` с валидным телом, система должна создать задачу с UUID и вернуть 201.
- **State-driven:** Пока задача существует в хранилище, система должна возвращать её данные при запросе по ID.
- **Optional feature:** Если поле `description` не указано при создании, система должна сохранить его как null.
- **Unwanted behavior:** Если `title` пустой или длиннее 255 символов, система должна вернуть 400 с описанием ошибки.
- **Unwanted behavior:** Если задача с указанным ID не найдена, система должна вернуть 404.

## Нерешённые вопросы

[RESOLVED CL-1] UUID-формат в URL: строгий UUID (8-4-4-4-12 hex). Не-UUID → 400.
[RESOLVED CL-2] PATCH не включён в v1. Только PUT для обновления.
[RESOLVED CL-3] DELETE возвращает 200 с JSON-телом удалённой задачи.
[RESOLVED CL-4] Формат времени: ISO-8601 UTC (`2024-01-15T10:30:00Z`).
[RESOLVED CL-5] Ошибки 400/404 возвращают структурированное JSON: `{"error", "message", "status", "timestamp"}`.

## Требования к безопасности (NFR-Security)

- **Auth:** Эндпоинты без аутентификации и авторизации (внутренний API). Это ограничение v1 — auth не является целью текущей спеки.
- **Input validation:** Обязательная валидация всех входных полей. `title` — обязательно, max 255 символов. `status` — только из допустимого набора. Защита от null/empty payload.
- **Injection:** In-memory хранилище не подвержено SQL-инъекциям. Валидация длины title защищает от潜在ного DoS через большие payload.
- **No secrets:** Секреты, токены и пароли не фигурируют в спецификации.

## Требования к приватности (NFR-Privacy)

- Задачи не содержат персональных данных (PII). Поля: title, description — текст, вводимый пользователем.
- In-memory хранилище не сохраняет данные между перезапусками приложения.

## Требования к производительности (NFR-Performance)

- **Throughput:** In-memory хранилище — O(1) для CRUD операций. Нецелесообразно замерять производительность на < 1000 элементов.
- **Latency:** Каждый endpoint должен отвечать < 100ms при in-memory данных (без учёта сетевого времени).
- **Ограничение:** Без подключения к БД количество задач ограничено доступной JVM-памятью.

## Требования к доступности (NFR-Availability)

- In-memory хранилище не обеспечивает persistence. Данные теряются при перезапуске приложения — это ожидаемое поведение v1.

## Требования к наблюдаемости (NFR-Observability)

- **Логирование:** Каждая операция CREATE, UPDATE, DELETE логирует ID задачи и операцию на INFO-уровне.
- **Ошибки:** 400 и 404 ответы логируются на WARN-уровне с описанием причины.
- **Метрики:** В v1 не добавляются кастомные метрики. Стандартные Spring Boot Actuator-метрики недоступны (не добавлена зависимость).
- **Трассировка:** В v1 не используется distributed tracing.
