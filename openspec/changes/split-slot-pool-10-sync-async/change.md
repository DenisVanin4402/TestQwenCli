## 1. Предложение

### Цель изменения

Сервис должен использовать фиксированный общий лимит из 10 слотов для обращений к лимитированному внешнему ресурсу. Этот лимит должен быть разделён на две независимые квоты: 5 слотов для sync-потока и 5 слотов для async-потока.

Изменение устраняет мягкую модель резерва sync-слотов, при которой async-поток мог занимать большую часть общего пула при сохранении минимального резерва. Новая модель делает SLA обоих режимов предсказуемым: sync-нагрузка не вытесняет async-dispatcher, а async backlog не занимает capacity, выделенную sync-запросам.

Свободные слоты одного потока не должны заимствоваться другим потоком. Если заняты все 5 sync-слотов, новый sync-запрос должен ожидать только sync-слот и завершаться `NO_SLOT_AVAILABLE` после timeout, даже если async-слоты свободны. Если заняты все 5 async-слотов, async dispatcher должен вернуть задачу в очередь без обращения к внешнему ресурсу и без увеличения числа upstream-попыток, даже если sync-слоты свободны.

### Инициатор

Пользовательское требование: сделать количество слотов равным 10 и разделить их поровну между sync- и async-потоками.

## 2. Бизнес-логика

### MODIFIED: 2.3.1. Лимит параллельных обращений через слоты

**Формулировка:** ЕСЛИ gateway принимает обращение к внешнему ресурсу, ТО он ДОЛЖЕН применять фиксированные квоты `syncSlots=5`, `asyncSlots=5`, `totalSlots=10`. Sync-запрос может занять только свободный sync-слот. Async dispatcher может занять только свободный async-слот. ИНАЧЕ sync-запрос ожидает sync-слот до `external-gateway.sync.wait-timeout-ms`, а async dispatcher возвращает задачу в `PENDING` без обращения к внешнему ресурсу.

Математические ограничения:

- `syncBusy <= 5`
- `asyncBusy <= 5`
- `syncBusy + asyncBusy <= 10`
- `syncSlots + asyncSlots = totalSlots = 10`

Свободные sync-слоты НИКОГДА не используются async-потоком. Свободные async-слоты НИКОГДА не используются sync-потоком.

**Пошаговый бизнес-псевдокод:**

1. Определить тип обращения: `SYNC` или `ASYNC`.
2. Если тип обращения `SYNC`, считать доступными только слоты категории `SYNC`.
3. Если тип обращения `ASYNC`, считать доступными только слоты категории `ASYNC`.
4. Для `SYNC` проверить, что число занятых sync-слотов меньше `5`.
5. Если свободный sync-слот есть, выдать lease категории `SYNC` с уникальным lease ID и временем истечения.
6. Если свободного sync-слота нет, зарегистрировать ожидание и повторять попытку до sync deadline.
7. Если sync deadline истёк, завершить sync-сценарий ошибкой `NO_SLOT_AVAILABLE`; внешний ресурс не вызывать.
8. Для `ASYNC` проверить, что число занятых async-слотов меньше `5`.
9. Если свободный async-слот есть, выдать lease категории `ASYNC` с уникальным lease ID и временем истечения.
10. Если свободного async-слота нет, вернуть задачу в очередь без увеличения `attempts`, без backoff и без вызова внешнего ресурса.
11. После завершения обращения освободить lease только при совпадении slot ID и lease ID.
12. Если lease не освобождён явно, reaper освобождает его после TTL без изменения категории слота.

Краевые случаи: пустой owner недопустим; конкурентные lease-операции не должны приводить к превышению квоты; округление не применяется, потому что квоты заданы целыми значениями `5` и `5`.

### MODIFIED

#### POST /v1/external/sync

- **Что именно меняется**: шаг получения слота и ошибочная ветка отсутствия слота.

**Шаги (было):**

1. Проверить тело запроса.
2. Попытаться получить sync-слот до истечения `external-gateway.sync.wait-timeout-ms`.
3. Если слот не получен, вернуть `429 NO_SLOT_AVAILABLE`.
4. Если слот получен, вызвать внешний ресурс.
5. Сохранить sync trace и освободить lease.

**Шаги (стало):**

1. Проверить тело запроса.
2. Попытаться получить только слот категории `SYNC` до истечения `external-gateway.sync.wait-timeout-ms`.
3. Если заняты все 5 sync-слотов до истечения timeout, вернуть `429 NO_SLOT_AVAILABLE`, даже если есть свободные async-слоты.
4. Если sync-слот получен, вызвать внешний ресурс.
5. Сохранить sync trace и освободить lease, сохранив категорию слота `SYNC`.

- **Изменение side-effects**: lease записывается и освобождается только в sync-пуле.
- **Изменение идемпотентности**: Нет изменений.
- **Обоснование**: sync SLA должен зависеть только от sync-квоты.

#### async dispatcher

- **Что именно меняется**: шаг получения слота после claim задачи.

**Шаги (было):**

1. Выбрать доступную `PENDING`-задачу.
2. Перевести её в `IN_PROGRESS`.
3. Получить async-слот с учётом общего пула и резерва sync-слотов.
4. Если слот недоступен, вернуть задачу в `PENDING` без вызова внешнего ресурса.
5. Если слот получен, вызвать внешний ресурс.
6. Сохранить результат или ошибку и освободить lease.

**Шаги (стало):**

1. Выбрать доступную `PENDING`-задачу.
2. Перевести её в `IN_PROGRESS`.
3. Получить только слот категории `ASYNC`.
4. Если заняты все 5 async-слотов, вернуть задачу в `PENDING`, не увеличивать `attempts`, не применять backoff и не вызывать внешний ресурс, даже если есть свободные sync-слоты.
5. Если async-слот получен, вызвать внешний ресурс.
6. Сохранить результат или ошибку и освободить lease, сохранив категорию слота `ASYNC`.

- **Изменение side-effects**: lease записывается и освобождается только в async-пуле.
- **Изменение идемпотентности**: Нет изменений.
- **Обоснование**: async throughput должен ограничиваться отдельной квотой и не занимать sync capacity.

#### slot lease reaper

- **Что именно меняется**: освобождение истёкшего lease.

**Шаги (было):**

1. Найти lease слотов, чей TTL истёк.
2. Освободить такие слоты.
3. При освобождении слота уведомить ожидающие sync-запросы, если включён режим ожидания через notification.

**Шаги (стало):**

1. Найти lease слотов, чей TTL истёк.
2. Освободить такие слоты без изменения категории `SYNC` или `ASYNC`.
3. Если освобождён слот категории `SYNC`, уведомить ожидающие sync-запросы, если включён режим ожидания через notification.
4. Если освобождён слот категории `ASYNC`, notification для sync waiters не требуется.

- **Изменение side-effects**: категория слота сохраняется после reaper-освобождения.
- **Изменение идемпотентности**: Нет изменений.
- **Обоснование**: reaper не должен разрушать разделение slot pool на sync и async.

## 12. Миграция

### Шаги миграции данных

1. Перед изменением slot pool дождаться отсутствия активных lease или остановить приём новых upstream-вызовов на время миграционного окна.
2. Привести `ext_slots` к целевому состоянию: ровно 10 строк, из них 5 строк с `kind=SYNC` и 5 строк с `kind=ASYNC`.
3. Для свободных слотов оставить `lease_id`, `owner`, `acquired_at`, `expires_at`, `task_id` пустыми.
4. После миграции проверить, что `syncBusy=0`, `asyncBusy=0`, `syncSlots=5`, `asyncSlots=5`, `totalSlots=10`.

### Обратная совместимость API

HTTP API не меняется. Клиенты могут продолжать использовать те же sync- и async-endpoints. Изменяется только capacity-поведение при конкуренции потоков.

### План отката

1. Дождаться отсутствия активных lease или остановить приём новых upstream-вызовов.
2. Вернуть прежние значения конфигурации slot pool.
3. Привести `ext_slots` к прежнему числу строк, не удаляя строки с активными lease.
4. Повторно включить обработку sync- и async-запросов.

| 16.6 | Slot lease освобождается обычным завершением обработки или reaper-ом | Слот становится свободным, но его категория `SYNC` или `ASYNC` сохраняется. |
| 16.7 | Gateway запускается с `total != 10`, `sync-limit != 5`, `async-limit != 5` или суммой квот, не равной total | Конфигурация признаётся невалидной, обработка upstream-вызовов не начинается. |
| 16.8 | Dashboard или monitoring snapshot запрашивает состояние slot pool | Snapshot показывает занятость и capacity отдельно для sync, async и total. |
| 16.9 | Метрики слотов публикуются после обработки sync- и async-нагрузки | `external_gateway_slots_busy` и `external_gateway_slot_utilization` доступны с разрезом `kind=sync`, `kind=async`, `kind=total`. |
